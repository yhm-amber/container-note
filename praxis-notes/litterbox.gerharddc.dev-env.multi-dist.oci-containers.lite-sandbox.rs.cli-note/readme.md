It is not intented to shield you from software that is known to be malicious so please do not run malware or untrusted software inside it deliberatly. Its only goal is to try and provide SOME damange limitation in the event that 3rd party software and/or code that you trust were to unexpectedly get compromised. // Litterbox 提供的隔离/沙盒机制有限，仍然存在许多漏洞和/或安全隐患。它并非旨在保护您免受已知恶意软件的侵害，因此请勿故意在其中运行恶意软件或不受信任的软件。它的唯一目标是在您信任的第三方软件和/或代码意外遭到入侵时，尽可能地限制其造成的损害。 /// By design, Litterbox comes with AT LEAST the following limitation/vulnerabilities: // Litterbox 的设计本身就存在至少以下限制/漏洞： /// - Everything running inside a Litterbox is running on top of your host kernel in the same way as normal applications. Thus, anything running inside the Litterbox could still exploit vulnerabilities in your host kernel to gain full access to your system. // Litterbox 内部运行的所有程序都像普通应用程序一样运行在宿主机内核之上。因此，Litterbox 内部运行的任何程序仍然可能利用宿主机内核中的漏洞来获取对系统的完全访问权限。 /// - Everything running inside a Litterbox has full access to your Wayland server in the same way as normal applications. Thus, anything running inside the Litterbox could still exploit vulnerabilities in your Wayland server to gain full access to your system. // Litterbox 内部运行的所有程序都像普通应用程序一样，可以完全访问您的 Wayland 服务器。因此，任何在 Litterbox 内部运行的程序仍然可以利用 Wayland 服务器中的漏洞来获取您系统的完全访问权限。 /// - Since applications running inside a Litterbox have normal access to your Wayland server, they have full access to things such as your clipboard so you should avoid copying any sensitive data around while you have a Litterbox running. // 由于在 Litterbox 中运行的应用程序可以正常访问您的 Wayland 服务器，因此它们可以完全访问剪贴板等内容，所以您应该避免在 Litterbox 运行时复制任何敏感数据。 /// - If you enable PipeWire support, everything running inside a Litterbox has full access to your PipeWire server. The goal of this server is to provide restricted access to SSH keys inside a Litterbox through a shared socket. Thus, anything running inside a Litterbox could still exploit vulnerabilities in this library to gain full access to your system. // Litterbox 托管了一个基于 russh 的 SSH 代理服务器。该服务器旨在通过共享套接字为 Litterbox 内部的 SSH 密钥提供受限访问。因此，任何在 Litterbox 内部运行的程序仍然可能利用该库中的漏洞来获取对您系统的完全访问权限。 /// - When you expose a device inside a Litterbox, you grant everything inside the Litterbox full access to that device. // 当你在猫砂盆内放置一个设备时，你就授予了猫砂盆内的所有设备对该设备的完全访问权限。 /// - Currently Litterbox has many external dependencies which unfortunately makes Litterbox itself vulnerable to supply chain attacks. A long-term goal is thus to reduce the number of external dependencies to a bare minimum. // 目前，Litterbox 存在诸多外部依赖项，这使得 Litterbox 本身容易受到供应链攻击。因此，长期目标是将外部依赖项的数量减少到最低限度。 /// N.B. it is again emphasised that Litterbox does not come with any warranties or guarantees. [site/work]: https://litterbox.work/ "Usage // 用法 /// - 1. Define // 1. 定义 ///: First you will need to define your Litterbox by running litterbox define LBX_NAME. This will prompt you to pick a template and will place a Dockerfile in your ~/Litterbox/definitions directory. The templates are a bit opinionated about what gets installed by default, so feel free to modify them! Please take note that (as described in the Dockerfile templates), anything you do inside the container’s home directory during the image build phase will “disappear” when the container runs. This is because a different directory on your host (in ~/Litterbox/homes) gets mounted over it at runtime. Thus, the Dockerfiles instead provide a script which gets run the first time that the container starts in order to set up the home directory. // 首先，您需要运行 litterbox define LBX_NAME 来定义您的 Litterbox。这将提示您选择一个模板，并将一个 Dockerfile 文件放置在您的 ~/Litterbox/definitions 目录中。这些模板对默认安装的内容有一定的预设，您可以随意修改它们！请注意（如 Dockerfile 模板中所述），在镜像构建阶段，您在容器主目录中所做的任何操作都会在容器运行时“消失”。这是因为在运行时，主机上的另一个目录（位于 ~/Litterbox/homes 中）会被挂载到容器主目录中。因此，Dockerfile 提供了一个脚本，该脚本会在容器首次启动时运行，以设置主目录。 /// - 2. Build // 2. 建造 ///: Then you will need to build your Litterbox by running litterbox build LBX_NAME. If you ever want to delete it again, simply run litterbox delete LBX_NAME. If you try to build a Litterbox that already exists, you will be offered the option to rebuild it or to do nothing. // 然后，您需要运行 litterbox build LBX_NAME 来构建您的猫砂盆。如果您以后想删除它，只需运行 litterbox delete LBX_NAME 即可。如果您尝试构建一个已存在的猫砂盆，系统会询问您是否要重建它，或者选择不执行任何操作。 ///; During the build process, you will be asked various questions related to how you want to configure this Litterbox. These primarily concern which non-default access you want to give this Litterbox (such as wether it should have access to PipeWire). These settings are stored at ~/Litterbox/LBX_NAME.ron and can be changed either by editing the file directly or by rebuilding the Litterbox and opting to change the settings. You will have to rebuild the Litterbox after changing the settings file for things to take effect though. // 在构建过程中，系统会询问您一些与配置此 Litterbox 相关的问题。这些问题主要涉及您希望授予此 Litterbox 哪些非默认访问权限（例如，是否允许其访问 PipeWire）。这些设置存储在 ~/Litterbox/LBX_NAME.ron 中，您可以通过直接编辑该文件或重新构建 Litterbox 并选择更改设置来更改它们。但是，您必须在更改设置文件后重新构建 Litterbox 才能使更改生效。 /// - 3. Enter // 3. 输入 ///: Finally you can then enter your Litterbox by running litterbox enter LBX_NAME. Once inside the Litterbox you can then start working on your projects! You can enter the same Litterbox multiple times from different terminals - all terminals share the same running container and this container will automatically stop when the last terminal exits. // 最后，您可以通过运行 litterbox enter LBX_NAME 进入您的 Litterbox。进入 Litterbox 后，您就可以开始处理您的项目了！您可以从不同的终端多次进入同一个 Litterbox——所有终端共享同一个运行中的容器，并且当最后一个终端退出时，该容器将自动停止。 /// - 4. Keys // 4. 钥匙 ///: If you want SSH keys to be available inside a Litterbox, simply run litterbox keys generate KEY_NAME to genererate a random key. You can then attach it to a Litterbox by running litterbox keys attach KEY_NAME LBX_NAME and detach it again using litterbox keys attach KEY_NAME. You can also view the public key by running litterbox keys print KEY_NAME. When a key is attached to a Litterbox, it is available through an SSH agent socket and each attempted interaction with the agent prompts a confirmation window to pop up. Also note that the keys are stored in ~/Litterbox/keys.ron and encrypted with a password that you chose. // 如果您希望在 Litterbox 中可用 SSH 密钥，只需运行 litterbox keys generate KEY_NAME 生成一个随机密钥。然后，您可以通过运行 litterbox keys attach KEY_NAME LBX_NAME 将其附加到 Litterbox，并使用 litterbox keys attach KEY_NAME 将其分离。您还可以通过运行 litterbox keys print KEY_NAME 查看公钥。密钥附加到 Litterbox 后，即可通过 SSH 代理套接字访问，每次尝试与代理交互时都会弹出确认窗口。另请注意，密钥存储在 ~/Litterbox/keys.ron 中，并使用您选择的密码进行加密。 /// - 5. Devices // 5. 设备 ///: If you ever need to make a device (such as a virtual serial port) available inside a Litterbox, simply run litterbox device LBX_NAME DEVICE_PATH. This will make the device available inside the Litterbox by creating a device node inside its home directory. To remove the device again later, simply delete this file that got created. Please note that the device node corresponds to a device using its device number and not some higher level identifier. Thus, if you for instance unplug the device and plug in a new device of the same type, the device node will now point to the new device. So be careful what you expose inside the Litterbox! // 如果您需要在 Litterbox 中启用某个设备（例如虚拟串口），只需运行 litterbox device LBX_NAME DEVICE_PATH 即可。该命令会在其主目录中创建一个设备节点，从而使该设备在 Litterbox 中可用。之后要移除该设备，只需删除创建的这个文件即可。请注意，设备节点指向的是设备的设备编号，而不是更高级别的标识符。因此，如果您拔下旧设备并插入一个相同类型的新设备，设备节点将指向新设备。所以，请谨慎在 Litterbox 中公开任何设备！ /// Comparison to alternatives /// 与替代方案的比较 /// - Full Virtual Machine // 完整虚拟机 ///: Even though good isolation can be achieved using a virtual machine, the idea with Litterbox is to provide decent isolation coupled with more convenience and less overhead. Litterbox runs everything on top of your host Linux kernel (thereby reducing overhead) and inside a folder that exists directly on the host (thereby making it simpler to share files). Furthermore, Litterbox allows applications to connect directly to the Wayland server on your host system which means that applications running inside the Litterbox are graphically composed just like normal applications and seamlessly have access to things like your clipboard (so be careful what you put in there). // 尽管使用虚拟机也能实现良好的隔离，但 Litterbox 的设计理念是在提供良好隔离的同时，兼顾便捷性和更低的开销。Litterbox 的所有程序都运行在主机 Linux 内核之上（从而降低开销），并且位于主机上的一个独立文件夹中（从而简化了文件共享）。此外，Litterbox 允许应用程序直接连接到主机系统上的 Wayland 服务器，这意味着在 Litterbox 中运行的应用程序可以像普通应用程序一样进行图形化操作，并且可以无缝访问剪贴板等内容（因此请谨慎在剪贴板中输入内容）。 ///; N.B. copy and pasting files to/from the Litterbox currently won’t work as expected in many cases since file paths inside and outside the Litterbox are different. Copying data rather than paths should work as normal though. // 注意：目前在 Litterbox 内外复制粘贴文件在很多情况下无法正常工作，因为 Litterbox 内外的文件路径不同。但复制数据（而非路径）应该可以正常工作。 /// - DevContainers // 开发容器 ///: Litterbox is very similar to DevContainers in that is uses Dockerfiles and containers to create a repeatable and somewhat isolated environment for a development project. A drawback with DevContainers though is that they are intended to be driven by an IDE and therefore require deep IDE integration to work properly. In practice this means that they are only really useable through VSCode (and maybe a few others). Litterbox tries to take a much more flexible approach in that it encourages you to instead run your entire IDE inside the container together with you project(s). This has the advantage that your IDE needs no knowledge of Litterbox and that your host system is also isolated from the IDE and any extensions that might be running inside it. Furthermore, you can easily develop multiple projects together inside a single Litterbox if you like since there isn’t the same strong connection between a single code repository and a single DevContainer. // Litterbox 与 DevContainers 非常相似，它们都使用 Dockerfile 和容器为开发项目创建可重复且相对隔离的环境。然而，DevContainers 的一个缺点是它们需要 IDE 驱动，因此需要与 IDE 深度集成才能正常工作。实际上，这意味着它们只能通过 VSCode（或许还有其他一些 IDE）使用。Litterbox 则采用了一种更加灵活的方法，它鼓励用户将整个 IDE 与项目一起运行在容器内。这样做的好处是，IDE 无需了解 Litterbox 的任何信息，并且宿主机系统也与 IDE 及其内部运行的任何扩展程序完全隔离。此外，由于单个代码仓库与单个 DevContainer 之间没有像 DevContainer 那样紧密的关联，因此您可以轻松地在单个 Litterbox 中同时开发多个项目。 /// - Distrobox ///: Litterbox is most similar to Distrobox in terms of its design and functionality. The primary difference is that Distrobox does not aim to provide any isolation/sandboxing at all whereas Litterbox has a strong emphasis on providing it. Distrobox avoids sandboxing in order to provide more seamless integration between applications running inside the Distrobox and the host system. It tries to solve the problem of running software intended for a different distro as if it is running natively. Litterbox instead sacrificies much of the convenience that Distrobox provides in exchange for some isolation/sandboxing capabilities. // Litterbox 在设计和功能上与 Distrobox 最为相似。主要区别在于 Distrobox 完全不提供任何隔离/沙盒机制，而 Litterbox 则非常注重提供这种机制。Distrobox 避免使用沙盒机制，是为了在 Distrobox 内部运行的应用程序与宿主系统之间实现更无缝的集成。它试图解决将为不同发行版设计的软件运行得如同原生软件一样的问题。而 Litterbox 则牺牲了 Distrobox 提供的许多便利性，以换取一定的隔离/沙盒功能。 /// Stability // 稳定性 /// Litterbox is still early in its development lifecycle and not particularly stable yet. Litterbox does not follow proper “semver” yet since we want to reserve 1.0.0 to signify a decent level of stability. Currently, (in the context of a version specified as x.y.z), changes in y will indicate breaking changes. It is therefore important to keep track of release notes for breaking changes when updating as you might have to make changes to your Litterbox definitions and rebuild them. // Litterbox 目前仍处于早期开发阶段，稳定性尚待提高。由于我们希望将 1.0.0 版本保留为较为稳定的版本号，因此 Litterbox 目前尚未遵循标准的“语义化版本控制”（semver）。目前（在版本号为 xyz 情况下）， y 的更改将表示存在重大变更。因此，在更新时务必关注版本说明，因为您可能需要修改 Litterbox 定义并重新构建它们。"
