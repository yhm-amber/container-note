"What is Reactive Programming?"“什么是响应式编程？”

There are plenty of bad explanations and definitions out there on the internet. Wikipedia is too generic and theoretical as usual. Stackoverflow's canonical answer is obviously not suitable for newcomers. Reactive Manifesto sounds like the kind of thing you show to your project manager or the businessmen at your company. Microsoft's Rx terminology "Rx = Observables + LINQ + Schedulers" is so heavy and Microsoftish that most of us are left confused. Terms like "reactive" and "propagation of change" don't convey anything specifically different to what your typical MV* and favorite language already does. Of course my framework views react to the models. Of course change is propagated. If it wouldn't, nothing would be rendered.互联网上有很多错误的解释和定义。维基百科像往常一样过于笼统和理论化。 Stackoverflow 的规范答案显然不适合新手。 Reactive Manifesto 听起来像是您向项目经理或公司的商人展示的那种东西。微软的 Rx 术语“Rx = Observables + LINQ + Schedulers”是如此沉重和微软化，以至于我们大多数人都感到困惑。诸如“反应性”和“变化的传播”之类的术语与您的典型 MV* 和最喜欢的语言已经做的没有任何特别不同。当然，我的框架视图会对模型做出反应。当然会传播变化。如果它不会，则不会呈现任何内容。

So let's cut the bullshit.所以让我们减少废话。

Reactive programming is programming with asynchronous data streams.响应式编程是使用异步数据流进行编程。

In a way, this isn't anything new. Event buses or your typical click events are really an asynchronous event stream, on which you can observe and do some side effects. Reactive is that idea on steroids. You are able to create data streams of anything, not just from click and hover events. Streams are cheap and ubiquitous, anything can be a stream: variables, user inputs, properties, caches, data structures, etc. For example, imagine your Twitter feed would be a data stream in the same fashion that click events are. You can listen to that stream and react accordingly.在某种程度上，这并不是什么新鲜事。事件总线或您典型的点击事件实际上是一个异步事件流，您可以在其上观察并执行一些副作用。反应性是类固醇的想法。您可以创建任何内容的数据流，而不仅仅是点击和悬停事件。流很便宜而且无处不在，任何东西都可以是流：变量、用户输入、属性、缓存、数据结构等。例如，假设您的 Twitter 提要是一个数据流，其方式与点击事件相同。您可以收听该流并做出相应的反应。

On top of that, you are given an amazing toolbox of functions to combine, create and filter any of those streams. That's where the "functional" magic kicks in. A stream can be used as an input to another one. Even multiple streams can be used as inputs to another stream. You can merge two streams. You can filter a stream to get another one that has only those events you are interested in. You can map data values from one stream to another new one.最重要的是，您将获得一个惊人的功能工具箱，可以组合、创建和过滤任何这些流。这就是“功能性”魔法发挥作用的地方。一个流可以用作另一个流的输入。甚至多个流也可以用作另一个流的输入。您可以合并两个流。您可以过滤一个流以获取仅包含您感兴趣的那些事件的另一个流。您可以将数据值从一个流映射到另一个新流。

If streams are so central to Reactive, let's take a careful look at them, starting with our familiar "clicks on a button" event stream.如果流对于 Reactive 如此重要，那么让我们仔细看看它们，从我们熟悉的“点击按钮”事件流开始。

A stream is a sequence of ongoing events ordered in time. It can emit three different things: a value (of some type), an error, or a "completed" signal. Consider that the "completed" takes place, for instance, when the current window or view containing that button is closed.流是按时间排序的一系列正在进行的事件。它可以发出三种不同的东西：一个值（某种类型）、一个错误或一个“完成”信号。考虑到“完成”发生，例如，当包含该按钮的当前窗口或视图关闭时。

We capture these emitted events only asynchronously, by defining a function that will execute when a value is emitted, another function when an error is emitted, and another function when 'completed' is emitted. Sometimes these last two can be omitted and you can just focus on defining the function for values. The "listening" to the stream is called subscribing. The functions we are defining are observers. The stream is the subject (or "observable") being observed. This is precisely the Observer Design Pattern.我们仅异步捕获这些发出的事件，方法是定义一个将在发出值时执行的函数，另一个在发出错误时执行的函数，以及另一个在发出“完成”时执行的函数。有时这最后两个可以省略，您可以只专注于定义值的函数。对流的“收听”称为订阅。我们定义的函数是观察者。流是被观察的主题（或“可观察的”）。这正是观察者设计模式。

An alternative way of drawing that diagram is with ASCII, which we will use in some parts of this tutorial:绘制该图的另一种方法是使用 ASCII，我们将在本教程的某些部分中使用它：

--a---b-c---d---X---|->

a, b, c, d are emitted values
X is an error
| is the 'completed' signal
---> is the timeline

Since this feels so familiar already, and I don't want you to get bored, let's do something new: we are going to create new click event streams transformed out of the original click event stream.由于这感觉已经很熟悉了，而且我不想让您觉得无聊，所以让我们做一些新的事情：我们将创建从原始点击事件流转换而来的新点击事件流。

First, let's make a counter stream that indicates how many times a button was clicked. In common Reactive libraries, each stream has many functions attached to it, such as map, filter, scan, etc. When you call one of these functions, such as clickStream.map(f), it returns a new stream based on the click stream. It does not modify the original click stream in any way. This is a property called immutability, and it goes together with Reactive streams just like pancakes are good with syrup. That allows us to chain functions like clickStream.map(f).scan(g):首先，让我们制作一个计数器流，指示单击按钮的次数。在常见的 Reactive 库中，每个流都附加了很多函数，例如 map、filter、scan 等。当您调用其中一个函数时，例如 clickStream.map(f)，它会根据点击返回一个新的流溪流。它不会以任何方式修改原始点击流。这是一种称为不变性的属性，它与 Reactive 流一起使用，就像煎饼搭配糖浆一样。这允许我们链接像 clickStream.map(f).scan(g) 这样的函数：

  clickStream: ---c----c--c----c------c-->
               vvvvv map(c becomes 1) vvvv
               ---1----1--1----1------1-->
               vvvvvvvvv scan(+) vvvvvvvvv
counterStream: ---1----2--3----4------5-->

The map(f) function replaces (into the new stream) each emitted value according to a function f you provide. In our case, we mapped to the number 1 on each click. The scan(g) function aggregates all previous values on the stream, producing value x = g(accumulated, current), where g was simply the add function in this example. Then, counterStream emits the total number of clicks whenever a click happens.map(f) 函数根据您提供的函数 f 替换（到新流中）每个发出的值。在我们的例子中，我们在每次点击时映射到数字 1。 scan(g) 函数聚合流中所有先前的值，生成值 x = g(accumulated, current)，其中 g 只是本例中的 add 函数。然后，每当发生点击时，counterStream 就会发出点击总数。

To show the real power of Reactive, let's just say that you want to have a stream of "double click" events. To make it even more interesting, let's say we want the new stream to consider triple clicks as double clicks, or in general, multiple clicks (two or more). Take a deep breath and imagine how you would do that in a traditional imperative and stateful fashion. I bet it sounds fairly nasty and involves some variables to keep state and some fiddling with time intervals.为了展示 Reactive 的真正力量，假设您想要一个“双击”事件流。为了让它更有趣，假设我们希望新流将三次点击视为双击，或者一般来说，多次点击（两次或更多次）。深吸一口气，想象一下你将如何以传统的命令式和有状态的方式做到这一点。我敢打赌这听起来相当讨厌，并且涉及一些变量来保持状态和一些摆弄时间间隔。

Well, in Reactive it's pretty simple. In fact, the logic is just 4 lines of code. But let's ignore code for now. Thinking in diagrams is the best way to understand and build streams, whether you're a beginner or an expert.好吧，在 Reactive 中它非常简单。事实上，逻辑只是 4 行代码。但是让我们暂时忽略代码。无论您是初学者还是专家，用图表思考是理解和构建流的最佳方式。

Grey boxes are functions transforming one stream into another. First we accumulate clicks in lists, whenever 250 milliseconds of "event silence" has happened (that's what buffer(stream.throttle(250ms)) does, in a nutshell. Don't worry about understanding the details at this point, we are just demoing Reactive for now). The result is a stream of lists, from which we apply map() to map each list to an integer matching the length of that list. Finally, we ignore 1 integers using the filter(x >= 2) function. That's it: 3 operations to produce our intended stream. We can then subscribe ("listen") to it to react accordingly how we wish.灰盒是将一个流转换为另一个流的函数。首先，我们在列表中累积点击次数，每当 250 毫秒的“事件沉默”发生时（这就是 buffer(stream.throttle(250ms)) 所做的，简而言之。此时不要担心理解细节，我们只是现在演示 Reactive）。结果是列表流，我们从中应用 map() 将每个列表映射到与该列表长度匹配的整数。最后，我们使用 filter(x >= 2) 函数忽略 1 个整数。就是这样：3 个操作来产生我们想要的流。然后我们可以订阅（“收听”）它以根据我们的意愿做出相应的反应。

I hope you enjoy the beauty of this approach. This example is just the tip of the iceberg: you can apply the same operations on different kinds of streams, for instance, on a stream of API responses; on the other hand, there are many other functions available.我希望你喜欢这种方法的美妙之处。这个例子只是冰山一角：您可以对不同类型的流应用相同的操作，例如，对 API 响应流；另一方面，还有许多其他功能可用。

"Why should I consider adopting RP?"“我为什么要考虑采用RP？”

Reactive Programming raises the level of abstraction of your code so you can focus on the interdependence of events that define the business logic, rather than having to constantly fiddle with a large amount of implementation details. Code in RP will likely be more concise.响应式编程提高了代码的抽象级别，因此您可以专注于定义业务逻辑的事件的相互依赖性，而不必不断摆弄大量的实现细节。 RP 中的代码可能会更简洁。

The benefit is more evident in modern webapps and mobile apps that are highly interactive with a multitude of UI events related to data events. 10 years ago, interaction with web pages was basically about submitting a long form to the backend and performing simple rendering to the frontend. Apps have evolved to be more real-time: modifying a single form field can automatically trigger a save to the backend, "likes" to some content can be reflected in real time to other connected users, and so forth.在与与数据事件相关的大量 UI 事件高度交互的现代 Web 应用程序和移动应用程序中，优势更为明显。 10年前，网页交互基本上就是向后台提交一个长表单，向前台进行简单的渲染。应用程序已经发展得更加实时：修改单个表单字段可以自动触发保存到后端，对某些内容的“喜欢”可以实时反映给其他连接的用户，等等。

Apps nowadays have an abundancy of real-time events of every kind that enable a highly interactive experience to the user. We need tools for properly dealing with that, and Reactive Programming is an answer.现在的应用程序有大量的各种实时事件，可以为用户提供高度交互的体验。我们需要工具来正确处理这个问题，而响应式编程就是一个答案。

Request and response请求和响应

How do you approach this problem with Rx? Well, to start with, (almost) everything can be a stream. That's the Rx mantra. Let's start with the easiest feature: "on startup, load 3 accounts data from the API". There is nothing special here, this is simply about (1) doing a request, (2) getting a response, (3) rendering the response. So let's go ahead and represent our requests as a stream. At first this will feel like overkill, but we need to start from the basics, right?你如何用 Rx 解决这个问题？好吧，首先，（几乎）一切都可以是流。这就是 Rx 的口头禅。让我们从最简单的功能开始：“启动时，从 API 加载 3 个帐户数据”。这里没有什么特别的，这只是关于 (1) 发出请求，(2) 获得响应，(3) 呈现响应。因此，让我们继续将我们的请求表示为流。起初这会感觉有点矫枉过正，但我​​们需要从基础开始，对吧？

On startup we need to do only one request, so if we model it as a data stream, it will be a stream with only one emitted value. Later, we know we will have many requests happening, but for now, it is just one.在启动时我们只需要执行一个请求，因此如果我们将其建模为数据流，它将是一个只有一个发射值的流。后来，我们知道我们会有很多请求发生，但现在只有一个。

--a------|->

Where a is the string 'https://api.github.com/users'

This is a stream of URLs that we want to request. Whenever a request event happens, it tells us two things: when and what. "When" the request should be executed is when the event is emitted. And "what" should be requested is the value emitted: a string containing the URL.这是我们要请求的 URL 流。每当请求事件发生时，它都会告诉我们两件事：时间和内容。 “何时”应该执行请求是在发出事件时。应该请求的“什么”是发出的值：一个包含 URL 的字符串。

To create such stream with a single value is very simple in Rx*. The official terminology for a stream is "Observable", for the fact that it can be observed, but I find it to be a silly name, so I call it stream.在 Rx* 中创建这样的具有单个值的流非常简单。流的官方术语是“Observable”，因为它可以被观察到，但我觉得这个名字很傻，所以我称之为流。

var requestStream = Rx.Observable.just('https://api.github.com/users');

But now, that is just a stream of strings, doing no other operation, so we need to somehow make something happen when that value is emitted. That's done by subscribing to the stream.但是现在，这只是一个字符串流，没有做任何其他操作，所以我们需要以某种方式在发出该值时发生一些事情。这是通过订阅流来完成的。

requestStream.subscribe(function(requestUrl) {
  // execute the request
  jQuery.getJSON(requestUrl, function(responseData) {
    // ...
  });
}

Notice we are using a jQuery Ajax callback (which we assume you should know already) to handle the asynchronicity of the request operation. But wait a moment, Rx is for dealing with asynchronous data streams. Couldn't the response for that request be a stream containing the data arriving at some time in the future? Well, at a conceptual level, it sure looks like it, so let's try that.请注意，我们正在使用 jQuery Ajax 回调（我们假设您应该已经知道）来处理请求操作的异步性。但是等一下，Rx 是用来处理异步数据流的。该请求的响应不能是包含将来某个时间到达的数据的流吗？好吧，在概念层面上，它确实看起来像，所以让我们试试看。

requestStream.subscribe(function(requestUrl) {
  // execute the request
  var responseStream = Rx.Observable.create(function (observer) {
    jQuery.getJSON(requestUrl)
    .done(function(response) { observer.onNext(response); })
    .fail(function(jqXHR, status, error) { observer.onError(error); })
    .always(function() { observer.onCompleted(); });
  });
  
  responseStream.subscribe(function(response) {
    // do something with the response
  });
}

What Rx.Observable.create() does is create your own custom stream by explicitly informing each observer (or in other words, a "subscriber") about data events (onNext()) or errors (onError()). What we did was just wrap that jQuery Ajax Promise. Excuse me, does this mean that a Promise is an Observable?Rx.Observable.create() 所做的是通过显式通知每个观察者（或换句话说，“订阅者”）有关数据事件 (onNext()) 或错误 (onError()) 的信息来创建您自己的自定义流。我们所做的只是包装 jQuery Ajax Promise。请问，这是否意味着 Promise 是 Observable？

         

Yes.是的。

Observable is Promise++. In Rx you can easily convert a Promise to an Observable by doing var stream = Rx.Observable.fromPromise(promise), so let's use that. The only difference is that Observables are not Promises/A+ compliant, but conceptually there is no clash. A Promise is simply an Observable with one single emitted value. Rx streams go beyond promises by allowing many returned values.Observable 是 Promise++。在 Rx 中，您可以通过执行 var stream = Rx.Observable.fromPromise(promise) 轻松地将 Promise 转换为 Observable，所以让我们使用它。唯一的区别是 Observables 不符合 Promises/A+ 标准，但在概念上没有冲突。 Promise 只是一个具有单一发射值的 Observable。 Rx 流通过允许许多返回值超越了承诺。

This is pretty nice, and shows how Observables are at least as powerful as Promises. So if you believe the Promises hype, keep an eye on what Rx Observables are capable of.这非常好，并且展示了 Observables 至少与 Promises 一样强大。因此，如果您相信 Promises 的炒作，请留意 Rx Observables 的功能。

Now back to our example, if you were quick to notice, we have one subscribe() call inside another, which is somewhat akin to callback hell. Also, the creation of responseStream is dependent on requestStream. As you heard before, in Rx there are simple mechanisms for transforming and creating new streams out of others, so we should be doing that.现在回到我们的示例，如果您很快注意到，我们在另一个调用中有一个 subscribe() 调用，这有点类似于回调地狱。另外，responseStream的创建依赖于requestStream。正如您之前听说的那样，在 Rx 中有一些简单的机制可以从其他流中转换和创建新的流，所以我们应该这样做。

The one basic function that you should know by now is map(f), which takes each value of stream A, applies f() on it, and produces a value on stream B. If we do that to our request and response streams, we can map request URLs to response Promises (disguised as streams).您现在应该知道的一个基本函数是 map(f)，它获取流 A 的每个值，对其应用 f()，并在流 B 上产生一个值。如果我们对请求和响应流执行此操作，我们可以将请求 URL 映射到响应 Promise（伪装成流）。

var responseMetastream = requestStream
  .map(function(requestUrl) {
    return Rx.Observable.fromPromise(jQuery.getJSON(requestUrl));
  });

Then we will have created a beast called "metastream": a stream of streams. Don't panic yet. A metastream is a stream where each emitted value is yet another stream. You can think of it as pointers: each emitted value is a pointer to another stream. In our example, each request URL is mapped to a pointer to the promise stream containing the corresponding response.然后我们将创建一个名为“metastream”的野兽：流的流。不要惊慌。 Metastream 是一种流，其中每个发出的值都是另一个流。您可以将其视为指针：每个发出的值都是指向另一个流的指针。在我们的示例中，每个请求 URL 都映射到指向包含相应响应的承诺流的指针。

A metastream for responses looks confusing, and doesn't seem to help us at all. We just want a simple stream of responses, where each emitted value is a JSON object, not a 'Promise' of a JSON object. Say hi to Mr. Flatmap: a version of map() that "flattens" a metastream, by emitting on the "trunk" stream everything that will be emitted on "branch" streams. Flatmap is not a "fix" and metastreams are not a bug, these are really the tools for dealing with asynchronous responses in Rx.用于响应的元数据流看起来令人困惑，而且似乎对我们一点帮助也没有。我们只想要一个简单的响应流，其中每个发出的值都是一个 JSON 对象，而不是 JSON 对象的“Promise”。向 Flatmap 先生问好：map() 的一个版本，它通过在“主干”流上发射所有将在“分支”流上发射的东西来“扁平化”元数据流。 Flatmap 不是“修复”，Metastream 也不是 bug，这些确实是 Rx 中处理异步响应的工具。

var responseStream = requestStream
  .flatMap(function(requestUrl) {
    return Rx.Observable.fromPromise(jQuery.getJSON(requestUrl));
  });

Nice. And because the response stream is defined according to request stream, if we have later on more events happening on request stream, we will have the corresponding response events happening on response stream, as expected:好的。并且由于响应流是根据请求流定义的，如果我们稍后在请求流上发生更多事件，我们将在响应流上发生相应的响应事件，如预期的那样：

requestStream:  --a-----b--c------------|->
responseStream: -----A--------B-----C---|->

(lowercase is a request, uppercase is its response)

Now that we finally have a response stream, we can render the data we receive:现在我们终于有了一个响应流，我们可以渲染我们收到的数据：

responseStream.subscribe(function(response) {
  // render `response` to the DOM however you wish
});

Joining all the code until now, we have:加入所有代码到现在，我们有：

Gladly, RxJS comes with tools to make Observables from event listeners.每次单击刷新按钮时，请求流都应该发出一个新的 URL，以便我们可以获得新的响应。我们需要两件事：刷新按钮上的点击事件流（口头禅：任何东西都可以是流），我们需要更改请求流以依赖于刷新点击流。令人高兴的是，RxJS 附带了从事件监听器中生成 Observables 的工具。

Again, let's think in streams. When a 'close1' click event happens, we want to use the most recently emitted response on responseStream to get one random user from the list in the response. As such:同样，让我们​​考虑一下流。当“close1”点击事件发生时，我们希望使用 responseStream 上最近发出的响应来从响应列表中随机获取一个用户。像这样：

close1ClickStream: ------------c----->

In Rx* there is a combinator function called combineLatest that seems to do what we need. It takes two streams A and B as inputs, and whenever either stream emits a value, combineLatest joins the two most recently emitted values a and b from both streams and outputs a value c = f(x,y), where f is a function you define. It is better explained with a diagram:在 Rx* 中，有一个名为 combineLatest 的组合器函数似乎可以满足我们的需求。它以两个流 A 和 B 作为输入，每当其中一个流发出一个值时，combineLatest 将两个流中最近发出的两个值 a 和 b 连接起来，并输出一个值 c = f(x,y)，其中 f 是一个函数你定义。最好用一张图来解释：

stream A: --a-----------e--------i-------->
stream B: -----b----c--------d-------q---->
          vvvvvvvv combineLatest(f) vvvvvvv
          ----AB---AC--EC---ED--ID--IQ---->

where f is the uppercase function

We can apply combineLatest() on close1ClickStream and responseStream, so that whenever the close 1 button is clicked, we get the latest response emitted and produce a new value on suggestion1Stream. On the other hand, combineLatest() is symmetric: whenever a new response is emitted on responseStream, it will combine with the latest 'close 1' click to produce a new suggestion. That is interesting, because it allows us to simplify our previous code for suggestion1Stream, like this:我们可以在 close1ClickStream 和 responseStream 上应用 combineLatest()，这样每当单击关闭 1 按钮时，我们都会获得发出的最新响应并在 suggestion1Stream 上产生一个新值。另一方面，combineLatest() 是对称的：每当在 responseStream 上发出新的响应时，它将与最新的“close 1”点击相结合以产生新的建议。这很有趣，因为它允许我们简化之前的 suggestion1Stream 代码，如下所示：

This workflow has been effective in my experience.如果您认为 Rx* 将是响应式编程的首选库，请花点时间熟悉用于转换、组合和创建 Observable 的大量函数。如果您想了解流图中的这些功能，请查看 RxJava 非常有用的带有弹珠图的文档。每当您尝试做某事时遇到困难，请画出这些图表，思考它们，查看一长串功能，并进行更多思考。根据我的经验，此工作流程非常有效。

Once you start getting the hang of programming with Rx*, it is absolutely required to understand the concept of Cold vs Hot Observables. If you ignore this, it will come back and bite you brutally. You have been warned. Sharpen your skills further by learning real functional programming, and getting acquainted with issues such as side effects that affect Rx*.一旦您开始掌握使用 Rx* 进行编程的窍门，就绝对需要了解 Cold 与 Hot Observables 的概念。如果你忽视这一点，它会回来残忍地咬你一口。你被警告了。通过学习真正的函数式编程并熟悉影响 Rx* 的副作用等问题，进一步提高您的技能。

But Reactive Programming is not just Rx*. There is Bacon.js which is intuitive to work with, without the quirks you sometimes encounter in Rx*. The Elm Language lives in its own category: it's a Functional Reactive Programming language that compiles to JavaScript + HTML + CSS, and features a time travelling debugger. Pretty awesome.但响应式编程不仅仅是 Rx*。 Bacon.js 使用起来很直观，没有您有时在 Rx* 中遇到的怪癖。 Elm 语言属于自己的类别：它是一种函数式反应式编程语言，可编译为 JavaScript + HTML + CSS，并具有时间旅行调试器。非常棒。

Rx works great for event-heavy frontends and apps. But it is not just a client-side thing, it works great also in the backend and close to databases. In fact, RxJava is a key component for enabling server-side concurrency in Netflix's API. Rx is not a framework restricted to one specific type of application or language. It really is a paradigm that you can use when programming any event-driven software.Rx 非常适合事件密集型前端和应用程序。但这不仅仅是客户端的事情，它在后端和靠近数据库的地方也能很好地工作。事实上，RxJava 是 Netflix API 中启用服务器端并发的关键组件。 Rx 不是一个仅限于一种特定类型的应用程序或语言的框架。它确实是您在编写任何事件驱动软件时可以使用的范例。
