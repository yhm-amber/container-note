[src.pijul/nest]: https://nest.pijul.org/pijul/pijul "(GPL-2.0) The sound distributed version control system // 分布式版本控制系统 /// This is the repository of Pijul, a sound and fast distributed version control system based on a mathematical theory of asynchronous work. // 这是 Pijul 的存储库，一个基于异步工作数学理论的可靠且快速的分布式版本控制系统。 /// Channels are a way to maintain two related versions of a repository in the same place (a bit like branches in Git). // 频道是一种在同一位置维护一个存储库的两个相关版本的方式（有点像 Git 中的分支）。 /// Formally, a channel is a pointer to a set of changes (the state of a channel is a set of changes). // 从技术上讲，一个频道是指向一组更改的指针（频道的状态是一组更改）。 /// However, channels are different from Git branches, and do not serve the same purpose. In Pijul, independent changes commute, which means that in many cases where branches are used in Git, there is no need to create a channel in Pijul. // 然而，频道与 Git 分支不同，并且不服务于相同的目的。在 Pijul 中，独立的更改可以交换，这意味着在 Git 中使用分支的许多情况下，在 Pijul 中不需要创建频道。 /// The main differences with Git branches are: // 与 Git 分支的主要区别是： /// - The identity of a change doesn't depend on the branch it is on, or in other words, rebase and merge are the same operation in Pijul. // 一个变更的身份不取决于它所在的分支，换句话说，在 Pijul 中，rebase 和 merge 是相同的操作。 /// - This implies that conflicts do not mysteriously come back after you solve them (which is what git rerere is for). // 这意味着在您解决冲突后，冲突不会神秘地再次出现（这就是 git rerere 的作用）。 /// - Also, conflicts are between changes, so the resolution of a conflict on one channel solves the same conflict in all other channels. // 此外，冲突发生在变更之间，因此在一个渠道上解决冲突可以解决所有其他渠道中的相同冲突。"
[site/org]: https://pijul.org/ "Pijul is a free and open source (GPL2) distributed version control system. Its distinctive feature is to be based on a theory of patches, while still being fast and scalable. This makes it easy to learn and use, without any compromise on power or features. // Pijul 是一个免费且开源（GPL2）的分布式版本控制系统。其独特之处在于基于补丁理论，同时仍然快速且可扩展。这使得它易于学习和使用，同时不牺牲功能或性能。 /// Why? // 为什么？ /// - Commutation // 交换律 ///: In Pijul, independent changes can be applied in any order without changing the result or the version's identifier. This makes Pijul significantly simpler than workflows using git rebase or hg transplant. Pijul has a branch-like feature called \"channels\", but these are not as important as in other systems. For example, so-called feature branches are often just changes in Pijul. Keeping your history clean is the default. // 在 Pijul 中，独立更改可以以任何顺序应用，而不会改变结果或版本标识符。这使得 Pijul 比使用 git rebase 或 hg transplant 的工作流程要简单得多。Pijul 有一个类似分支的功能，称为“频道”，但这些功能在其他系统中的重要性并不那么高。例如，所谓的特性分支在 Pijul 中通常只是更改。保持你的历史记录干净是默认设置。 /// - Merge correctness // 合并正确性 ///: Pijul guarantees a number of strong properties on merges. The most important one is that the order between lines is always preserved. This is unlike 3-way merge, which may sometimes shuffle lines around. When the order is unknown (for example in the case of concurrent edits), this is a conflict, which contrasts with systems with \"automatic\" or \"no conflicts\" merges. // Pijul 对合并保证了一系列强属性。其中最重要的是始终保留行之间的顺序。这与三路合并不同，三路合并有时可能会打乱行的顺序。当顺序未知时（例如在并发编辑的情况下），这是一个冲突，这与具有“自动”或“无冲突”合并的系统形成对比。 /// - First-class conflicts // 一级冲突 ///: In Pijul, conflicts are not modelled as a “failure to merge”, but rather as the standard case. Specifically, conflicts happen between two changes, and are solved by one change. The resolution change solves the conflict between the same two changes, no matter if other changes have been made concurrently. Once solved, conflicts never come back. // 在 Pijul 中，冲突不是被建模为“合并失败”，而是被视为标准情况。具体来说，冲突发生在两个变更之间，并由其中一个变更解决。解决变更会解决这两个变更之间的冲突，无论是否同时进行了其他变更。一旦解决，冲突将不再出现。 /// - Partial clones // 部分克隆 ///: Commutation makes it possible to clone only a small subset of a repository: indeed, one can only apply the changes related to that subset. Working on a partial clone produces changes that can readily be sent to the large repository. // 交换操作使得可以克隆仓库的一个小子集：实际上，只能应用与该子集相关的变更。在部分克隆上工作产生的变更可以方便地发送到大型仓库。"
