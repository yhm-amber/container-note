Distributed version control分布式版本控制

Software engineering tool

In software development, distributed version control (also known as distributed revision control) is a form of version control in which the complete codebase, including its full history, is mirrored on every developer's computer.[1] Compared to centralized version control, this enables automatic management branching and merging, speeds up most operations (except pushing and pulling), improves the ability to work offline, and does not rely on a single location for backups.[1][2][3] Git, the world's most popular version control system,[4] is a distributed version control system. 在软件开发中，分布式版本控制（也称为分布式修订控制）是一种版本控制形式，其中完整的代码库（包括其完整历史记录）会镜像到每个开发人员的计算机上。 [1]与集中式版本控制相比，这实现了自动管理分支和合并，加快了大多数操作（推和拉除外），提高了离线工作的能力，并且不依赖于单一位置进行备份。[1][2][ 3] Git 是世界上最流行的版本控制系统，[4] 是一种分布式版本控制系统。

In 2010, software development author Joel Spolsky described distributed version control systems as "possibly the biggest advance in software development technology in the [past] ten years".[2]2010 年，软件开发作者 Joel Spolsky 将分布式版本控制系统描述为“可能是 [过去] 十年中软件开发技术的最大进步”。 [2]

Distributed vs. centralized[edit]分布式与集中式[编辑]

Distributed version control systems (DVCS) use a peer-to-peer approach to version control, as opposed to the client–server approach of centralized systems. Distributed revision control synchronizes repositories by transferring patches from peer to peer. There is no single central version of the codebase; instead, each user has a working copy and the full change history. 分布式版本控制系统 (DVCS) 使用对等方法进行版本控制，这与集中式系统的客户端-服务器方法相反。分布式修订控制通过点对点传输补丁来同步存储库。代码库没有单一的中央版本；相反，每个用户都有一个工作副本和完整的更改历史记录。

Advantages of DVCS (compared with centralized systems) include: DVCS 的优势（与集中式系统相比）包括：

  • Allows users to work productively when not connected to a network.允许用户在未连接到网络时高效工作。
  • Common operations (such as commits, viewing history, and reverting changes) are faster for DVCS, because there is no need to communicate with a central server.[5] With DVCS, communication is necessary only when sharing changes among other peers.DVCS 的常见操作（如提交、查看历史记录和恢复更改）速度更快，因为不需要与中央服务器通信。 [5]使用 DVCS，仅当在其他对等方之间共享更改时才需要进行通信。
  • Allows private work, so users can use their changes even for early drafts they do not want to publish.[citation needed]允许私人工作，因此用户甚至可以将他们的更改用于他们不想发布的早期草稿。[需要引证]
  • Working copies effectively function as remote backups, which avoids relying on one physical machine as a single point of failure.[5]工作副本有效地充当远程备份，这避免了依赖一台物理机器作为单点故障。 [5]
  • Allows various development models to be used, such as using development branches or a Commander/Lieutenant model.[6]允许使用各种开发模型，例如使用开发分支或指挥官/中尉模型。 [6]
  • Permits centralized control of the "release version" of the project[citation needed]允许集中控制项目的“发布版本”[需要引证]
  • On FOSS software projects it is much easier to create a project fork from a project that is stalled because of leadership conflicts or design disagreements.在 FOSS 软件项目中，从由于领导冲突或设计分歧而停滞的项目创建项目分支要容易得多。

Disadvantages of DVCS (compared with centralized systems) include: DVCS 的缺点（与集中式系统相比）包括：

  • Initial checkout of a repository is slower as compared to checkout in a centralized version control system, because all branches and revision history are copied to the local machine by default.与集中式版本控制系统中的检出相比，存储库的初始检出速度较慢，因为默认情况下所有分支和修订历史都被复制到本地机器。
  • The lack of locking mechanisms that is part of most centralized VCS and still plays an important role when it comes to non-mergeable binary files such as graphic assets or too complex single file binary or XML packages (e.g. office documents, PowerBI files, SQL Server Data Tools BI packages, etc.).[citation needed]缺少锁定机制，这是大多数集中式 VCS 的一部分，并且在涉及不可合并的二进制文件（例如图形资产）或过于复杂的单个文件二进制文件或 XML 包（例如办公文档、PowerBI 文件、SQL Server数据工具 BI 包等）。[需要引用]
  • Additional storage required for every user to have a complete copy of the complete codebase history.[7]每个用户都需要额外的存储空间才能拥有完整代码库历史记录的完整副本。 [7]
  • Increased exposure of the code base since every participant has a locally vulnerable copy.[citation needed]由于每个参与者都有一个本地易受攻击的副本，因此增加了代码库的曝光率。[需要引证]

Some originally centralized systems now offer some distributed features. For example, Subversion is able to do many operations with no network.[8] Team Foundation Server and Visual Studio Team Services now host centralized and distributed version control repositories via hosting Git. 一些原本集中的系统现在提供了一些分布式的特性。例如，Subversion 能够在没有网络的情况下执行许多操作。 [8] Team Foundation Server 和 Visual Studio Team Services 现在通过托管 Git 托管集中式和分布式版本控制存储库。

Similarly, some distributed systems now offer features that mitigate the issues of checkout times and storage costs, such as the Virtual File System for Git developed by Microsoft to work with very large codebases,[9] which exposes a virtual file system that downloads files to local storage only as they are needed. 同样，一些分布式系统现在提供的功能可以减轻结帐时间和存储成本的问题，例如 Microsoft 开发的用于 Git 的虚拟文件系统，用于处理非常大的代码库，[9] 它公开了一个虚拟文件系统，可以将文件下载到仅在需要时才进行本地存储。

Work model[edit]工作模式[编辑]

The distributed model is generally better suited for large projects with partly independent developers, such as the Linux kernel project, because developers can work independently and submit their changes for merge (or rejection). The distributed model flexibly allows adopting custom source code contribution workflows. The integrator workflow is the most widely used. In the centralized model, developers must serialize their work, to avoid problems with different versions. 分布式模型通常更适合具有部分独立开发人员的大型项目，例如 Linux 内核项目，因为开发人员可以独立工作并提交他们的更改以进行合并（或拒绝）。分布式模型灵活地允许采用自定义源代码贡献工作流。集成商工作流是使用最广泛的。在集中式模型中，开发人员必须将他们的工作序列化，以避免不同版本的问题。

Central and branch repositories[edit]中央和分支仓库[编辑]

In a truly distributed project, such as Linux, every contributor maintains their own version of the project, with different contributors hosting their own respective versions and pulling in changes from other users as needed, resulting in a general consensus emerging from multiple different nodes. This also makes the process of "forking" easy, as all that is required is one contributor stop accepting pull requests from other contributors and letting the codebases gradually grow apart. 在一个真正的分布式项目中，比如 Linux，每个贡献者维护他们自己的项目版本，不同的贡献者托管他们各自的版本，并根据需要从其他用户那里拉取更改，从而从多个不同的节点产生普遍的共识。这也使“分叉”过程变得容易，因为所需要的只是一个贡献者停止接受来自其他贡献者的拉取请求并让代码库逐渐分离。

This arrangement, however, can be difficult to maintain, resulting in many projects choosing to shift to a paradigm in which one contributor is the universal "upstream", a repository from whom changes are almost always pulled. Under this paradigm, development is somewhat recentralized, as every project now has a central repository that is informally considered as the official repository, managed by the project maintainers collectively. While distributed version control systems make it easy for new developers to "clone" a copy of any other contributor's repository, in a central model, new developers always clone the central repository to create identical local copies of the code base. Under this system, code changes in the central repository are periodically synchronized with the local repository, and once the development is done, the change should be integrated into the central repository as soon as possible. 然而，这种安排可能难以维护，导致许多项目选择转向一种范式，其中一个贡献者是通用的“上游”，一个几乎总是从中提取更改的存储库。在这种范式下，开发有点重新集中化，因为每个项目现在都有一个中央存储库，非正式地被认为是官方存储库，由项目维护者集体管理。虽然分布式版本控制系统使新开发人员可以轻松地“克隆”任何其他贡献者存储库的副本，但在中央模型中，新开发人员总是克隆中央存储库以创建代码库的相同本地副本。在该系统下，中央仓库中的代码变更会定期与本地仓库同步，​​一旦开发完成，变更应尽快集成到中央仓库中。

Organizations utilizing this centralize pattern often choose to host the central repository on a third party service like GitHub, which offers not only more reliable uptime than self-hosted repositories, but can also add centralized features like issue trackers and continuous integration. 使用这种集中模式的组织通常选择在 GitHub 等第三方服务上托管中央存储库，它不仅提供比自托管存储库更可靠的正常运行时间，而且还可以添加集中功能，如问题跟踪器和持续集成。

Pull requests[edit]拉取请求[编辑]

Contributions to a source code repository that uses a distributed version control system are commonly made by means of a pull request, also known as a merge request.[10] The contributor requests that the project maintainer pulls the source code change, hence the name "pull request". The maintainer has to merge the pull request if the contribution should become part of the source base.[11]对使用分布式版本控制系统的源代码存储库的贡献通常通过拉取请求（也称为合并请求）进行。 [10]贡献者请求项目维护者拉取源代码更改，因此得名“拉取请求”。如果贡献应该成为源库的一部分，维护者必须合并拉取请求。 [11]

The developer creates a pull request to notify maintainers of a new change; a comment thread is associated with each pull request. This allows for focused discussion of code changes. Submitted pull requests are visible to anyone with repository access. A pull request can be accepted or rejected by maintainers.[12]开发人员创建拉取请求以通知维护人员新的更改；评论线程与每个拉取请求相关联。这允许集中讨论代码更改。任何具有存储库访问权限的人都可以看到提交的拉取请求。维护者可以接受或拒绝拉取请求。 [12]

Once the pull request is reviewed and approved, it is merged into the repository. Depending on the established workflow, the code may need to be tested before being included into official release. Therefore, some projects contain a special branch for merging untested pull requests.[11][13] Other projects run an automated test suite on every pull request, using a continuous integration tool such as Travis CI, and the reviewer checks that any new code has appropriate test coverage. 一旦拉取请求得到审查和批准，它就会被合并到存储库中。根据既定的工作流程，代码可能需要在正式发布之前进行测试。因此，一些项目包含一个特殊的分支，用于合并未经测试的拉取请求。[11][13]其他项目使用持续集成工具（如 Travis CI）对每个拉取请求运行自动化测试套件，审查者检查任何新代码是否具有适当的测试覆盖率。

History[edit]历史[编辑]

The first open-source DVCS systems included Arch, Monotone, and Darcs. However, open source DVCSs were never very popular until the release of Git and Mercurial. 第一个开源 DVCS 系统包括 Arch、Monotone 和 Darcs。然而，在 Git 和 Mercurial 发布之前，开源 DVCS 从未流行过。

BitKeeper was used in the development of the Linux kernel from 2002 to 2005.[14] The development of Git, now the world's most popular version control system,[4] was prompted by the decision of the company that made BitKeeper to rescind the free license that Linus Torvalds and some other Linux kernel developers had previously taken advantage of.[14]从 2002 年到 2005 年，BitKeeper 被用于 Linux 内核的开发。 [14] Git 的开发，现在是世界上最流行的版本控制系统，[4] 是由于公司的决定促使 BitKeeper 取消 Linus Torvalds 和其他一些 Linux 内核开发人员以前利用的免费许可证。 [14 ]
