Using Sentinel Values 使用哨兵值

This one is a pet peeve of mine.这是我的一个小毛病。

In most C-based languages (C, C#, Java, etc.), the way you indicate whether something failed or couldn’t be found is by returning a “special” value. For example, C#’s String.IndexOf() method will scan an array for a particular element and return its index. Returning -1 if nothing is found.在大多数基于 C 的语言（C 、 C# 、 Java 等）中，表明某物是否失败或无法找到的方式是返回一个 “特殊” 值。例如， C# 的 String.IndexOf() 方法将扫描一个数组中的某个元素，并返回其索引。如果没有找到，则返回 -1 。

That leads to code like this:这就导致了这样的代码：

string sentence = "The fox jumps over the dog";

int index = sentence.IndexOf("fox");

if (index != -1)
{
  string wordsAfterFox = sentence.SubString(index);
  Console.WriteLine(wordsAfterFox);
}

You see this sort of “use a sentinel value to indicate something special” practice all the time. Other sentinel values you might find in the wild are "", or null (someone once referred to this as their “billion-dollar mistake”).你经常会看到这种 “用一个哨兵值来表示一些特殊的东西” 的做法。你可能在野外发现的其他哨兵值是 "" ，或 null （这还曾诞生过 “十亿美元错误” ）。

The general reason why this is a bad idea is that there is absolutely nothing to stop you from forgetting that check. That means you can accidentally crash your application with one misplaced assumption or when the code generating the sentinel is far away from the code using it.这是一个坏主意的一般原因是，绝对没有什么可以阻止你忘记这个检查。这意味着你可能会因为一个错误的假设，或者当生成哨兵的代码远离使用它的代码时，而意外地使你的应用程序崩溃。

We can do a lot better in Rust, though. Just use Option!不过在 Rust 中我们可以做得更好。只要使用 Option !

By design, there is no way to get the underlying value without dealing with the possibility that your Option may be None. This is enforced by the compiler at compile time, meaning code that forgets to check won’t even compile.根据设计，如果不处理你的 Option 可能是 None 的问题，就没有办法获得基础值。这是编译器在编译时强制执行的，这意味着忘记检查的代码甚至不会被编译。

let sentence = "The fox jumps over the dog";
let index = sentence.find("fox");

// let words_after_fox = &sentence[index..]; // Error: Can't index str with Option<usize>

if let Some(fox) = index {
  let words_after_fox = &sentence[fox..];
  println!("{}", words_after_fox);
}

Hungarian Notation 匈牙利语记号

Back in the 70’s, a naming convention called Hungarian Notation was developed by programmers writing in languages where variables are untyped or dynamically typed. It works by adding a mnemonic to the start of a name to indicate what it represents, for example the boolean visited variable might be called bVisited or the string name might be called strName.早在70年代，一种叫做 匈牙利符号 的命名方法被程序员开发出来，他们使用的语言中的变量是不类型的或动态类型的。它的工作原理是在名字的开头添加一个助记符，以表示它所代表的内容，例如，布尔型变量 visited 可能被称为 bVisited ，或者字符串 name 可能被称为 strName 。

You can still see this naming convention in languages Delphi where classes (types) start with T, fields start with F, arguments start with A, and so on.你仍然可以在Delphi语言中看到这种命名惯例，类（类型）以 T 开头，字段以 F 开头，参数以 A 开头，依此类推。

type
 TKeyValue = class
  private
    FKey: integer;
    FValue: TObject;
  public
    property Key: integer read FKey write FKey;
    property Value: TObject read FValue write FValue;
    function Frobnicate(ASomeArg: string): string;
  end;

C# also has a convention that all interfaces should start with I, meaning programmers coming to Rust from C# will sometimes prefix their traits with I as well.C# 也有一个惯例，即所有的接口都应该以 I 开头，这意味着从 C# 来到 Rust 的程序员有时也会在他们的 traits 前加上 I 。

trait IClone {
  fn clone(&self) -> Self;
}

In this case, just drop the leading I. Rust’s syntax guarantees that it just isn’t possible to confuse a trait for a normal type, so it isn’t helping anyone. This is in contrast with C# where interfaces and classes are largely interchangeable.在这种情况下，只要去掉前面的 I 就可以了。 Rust 的语法保证了不可能将 trait 与普通类型混淆，所以这对任何人都没有帮助。这与 C# 形成了对比，在 C# 中，接口和类在很大程度上是可以互换的。

This is also seen inside functions where people will conjure up new names for something as they convert it from one form to another. Often these names are silly or contrived, providing negligible additional information to the reader.这在函数中也可以看到，当人们将某样东西从一种形式转换到另一种形式时，就会为它想出新的名字。这些名字通常都很傻，或者是被设计出来的，给读者提供的额外信息几乎可以忽略不计。

let account_bytes: Vec<u8> = read_some_input();
let account_str = String::from_utf8(account_bytes)?;
let account: Account = account_str.parse()?;

I mean, if we’re calling String::from_utf8() we already know account_str will be a String so why add the _str suffix?我的意思是，如果我们调用 String::from_utf8() ，我们已经知道 account_str 将是一个 String ，那么为什么还要加上 _str 这个后缀呢？

Unlike a lot of other languages, Rust encourages shadowing variables when you are transforming them from one form to another, especially when the previous variable is no longer accessible (e.g. because it’s been moved).与其他很多语言不同， Rust 鼓励在你将变量从一种形式转化为另一种形式时对其进行阴影处理，特别是当之前的变量不再可以访问时（例如因为它被移动了）。

let account: Vec<u8> = read_some_input();
let account = String::from_utf8(account)?;
let account: Account = account.parse()?;

This is arguably superior because we can use the same name for the same concept.这可以说是一种优势，因为我们可以用同样的名字来表达同样的概念。

Other languages frown on shadowing because it can be easy to lose track of what type a variable contains (e.g. in a dynamically typed language like JavaScript) or you can introduce bugs where the programmer thinks a variable has one type but it actually contains something separate.其他语言不喜欢阴影，因为它很容易失去对一个变量所包含的类型的追踪（例如在像 JavaScript 这样的动态类型语言中），或者你可以引入错误，即程序员认为一个变量有一种类型，但它实际上包含一些不同的东西。

Neither of these is particularly relevant to a strongly typed language with move semantics like Rust, so you can use shadowing freely without worrying about shooting yourself in the foot.这两种情况对于像 Rust 这样具有移动语义的强类型语言来说都不是特别重要，所以你可以自由地使用阴影，而不用担心会给自己带来麻烦。

An Abundance of Rc<RefCell<T>> 这个 Rc<RefCell<T>> 出现得太多了

A common pattern in Object Oriented languages is to accept a reference to some object so you can call its methods later on.在面向对象的语言中，一个常见的模式是接受某个对象的引用，这样你就可以在以后调用其方法。

On its own there is nothing wrong with this, Dependency Injection is a very good thing to do, but unlike most OO languages Rust doesn’t have a garbage collector and has strong feelings on shared mutability.这本身并没有错，依赖注入是一件非常好的事情，但与大多数 OO 语言不同的是， Rust 没有垃圾收集器，而且对共享可变性有异常的警惕。

Perhaps this will be easier to understand with an example.也许用一个例子会更容易理解。

Say we are implementing a game where the player needs to beat up a bunch of monsters until they have inflicted a certain amount of damage (I dunno, maybe it’s for a quest or something).假设我们正在实现一个游戏，玩家需要殴打一群怪物，直到它们造成一定的伤害（我不知道，也许是为了一个任务或其他什么）。

We create a Monster class which has a health property and a takeDamage() method, and so we can keep track of how much damage has been inflicted we’ll let people provide callbacks that get called whenever the monster receives damage.我们创建一个 Monster 类，它有一个 health 属性和一个 takeDamage() 方法，因此我们可以跟踪已经造成了多少伤害，我们将让人们提供回调，每当怪物受到伤害时就会调用。

type OnReceivedDamage = (damageReceived: number) => void;

class Monster {
    health: number = 50;
    receivedDamage: OnReceivedDamage[] = [];

    takeDamage(amount: number) {
        amount = Math.min(this.health, amount);
        this.health -= amount;
        this.receivedDamage.forEach(cb => cb(amount));
    }

    on(event: "damaged", callback: OnReceivedDamage): void {
        this.receivedDamage.push(callback);
    }
}

Let’s also create a DamageCounter class which tracks how much damage we’ve inflicted and lets us know when that goal is reached.让我们也创建一个 DamageCounter 类，它可以跟踪我们造成了多少伤害，并让我们知道什么时候达到了这个目标。

class DamageCounter {
    damageInflicted: number = 0;

    reachedTargetDamage(): boolean {
        return this.damageInflicted > 100;
    }

    onDamageInflicted(amount: number) {
        this.damageInflicted += amount;
    }
}

Now we’ll create some monsters and keep inflicting a random amount of damage until the DamageCounter is happy.现在我们将创建一些怪物，并不断施加随机的伤害，直到 DamageCounter 满意为止。

const counter = new DamageCounter();

const monsters = [new Monster(), new Monster(), new Monster(), new Monster(), new Monster()];
monsters.forEach(m => m.on("damaged", amount => counter.onDamageInflicted(amount)));

while (!counter.reachedTargetDamage()) {
    // pick a random monster
    const index = Math.floor(Math.random()*monsters.length);
    const target = monsters[index];
    // then damage it a bit
    const damage = Math.round(Math.random() * 50);
    target.takeDamage(damage);

    console.log(`Monster ${index} received ${damage} damage`);
}

fn main() {

But this approach tends to get messy, especially when you are storing non-trivial things like a Rc<RefCell<Vec<Foo>>>> (or its multi-threaded cousin Arc<Mutex<Vec<Foo>>>>) inside structs 1.嗯 …… 这很有效。但这种方法往往会变得混乱，特别是当你在结构1内存储非微不足道的东西，如 Rc<RefCell<Vec<Foo>>>> （或其多线程的表弟 Arc<Mutex<Vec<Foo>>>> ）。

A much better approach is to change your API to not hold long-lived references to other objects. Depending on the situation, it might make sense to take a callback argument in the Monster::take_damage() method.一个更好的方法是改变你的 API ，不持有对其他对象的长期引用。根据不同的情况，在 Monster::take_damage() 方法中接受一个回调参数可能是有意义的。

struct Monster {
    health: u32,
}

impl Monster {
    fn take_damage(&mut self, amount: u32, on_damage_received: impl FnOnce(u32)) {
        let damage_received = cmp::min(self.health, amount);
        self.health -= damage_received;
        on_damage_received(damage_received);
    }
}

impl Default for Monster {
  fn default() -> Self { Monster { health: 100 } }
}

...

    damage_received: u32,
}

Using the Wrong Integer Type 使用错误的整数类型

Another hang-over from writing a lot of C is using the wrong integer type and getting frustrated because you need to cast to/from usize all the time.因编写大量 C 带来的另一个威胁，是使用错误的整数类型，并且你会在，非要不断地从 usize 转换或转换成它的过程中，真切地感受到悲伤和沮丧。

I’ve seen people run into this 我已（在野外）见人们掉到这种情况里so许 many多 times次 in the wild, especially when indexing.了，特别是在索引时。

The underlying problem is that C programmers are all taught to use int for indexing and for-loops, so when they come to Rust and they need to store a list of indices, the programmer will immediately reach for a Vec<i32>. They then get frustrated because Rust is quite strict when it comes to indexing and standard types like arrays, slices, and Vec can only be indexed using usize (the equivalent of size_t), meaning their code is cluttered with casts from i32 to usize and back again.潜在的问题是， C 语言程序员都被教导要使用 int 进行索引和 for 循环，所以当他们来到 Rust ，需要存储一个索引列表时，程序员会立即伸手去拿 Vec<i32> 。然后他们就会感到沮丧，因为 Rust 在索引方面相当严格，像数组、切片和 Vec 这样的标准类型只能使用 usize （相当于 size_t ）进行索引，这意味着他们的代码被迫要从 i32 转换到 usize 再转回去从而弄得很混乱。

There are a number of perfectly legitimate reasons for why Rust only allows indexing by usize:为什么 Rust 只允许用 usize 做索引，有很多完全合理的理由：

  • It doesn’t make sense to have a negative index (accessing items before the start of a slice is UB), so we can avoid an entire class of bugs by indexing with an unsigned integer有一个负的索引是没有意义的（在一个片断的开始之前访问项目是不合法的），所以我们可以通过用无符号的整数做索引来避免一整类的错误
  • A usize is defined to be an integer with the same size as a normal pointer, meaning the pointer arithmetic won’t have any hidden casts usize 被定义为一个整数，其大小与普通指针相同，这意味着指针运算不会有任何隐藏的转换。
  • The std::mem::size_of() and std::mem::align_of() functions return usize std::mem::size_of() 和 std::mem::align_of() 函数返回 usize 。

Of course, when stated this way the solution is clear. Choose the right integer type for your application but when you are doing things that eventually be used for indexing, that “right integer type” is probably usize.当然，这样说的话，解决方案就很清楚了。为你的应用选择正确的整数类型，但当你做的事情最终被用于索引时，这个 “正确的整数类型” 可能是 usize 。

Not Using Namespaces 不使用命名空间

A common practice in C is to prefix functions with the name of the library or module to help readers understand where it comes from and avoid duplicate symbol errors (e.g. rune_wasmer_runtime_load()).在 C 语言中，一个常见的做法是在函数前加上库或模块的名字，以帮助读者了解它的来源，避免重复符号错误（例如 rune_wasmer_runtime_load() ）。

Overusing Iterators 过度使用迭代器

Once you start drinking the Kool-Aid that is Rust’s iterators you can run into the opposite problem - when all you have is a hammer everything looks like a nail.一旦你开始觉得 Rust 迭代器是超酷的万全法宝，你就可能会遇到相反的问题 —— 当你拥有的只是一把锤子时，一切看起来都像钉子。

Other times your iterator-based solution is just unnecessarily complicated.其他时候，你基于迭代器的解决方案只是不必要的复杂。

As an example, have a look at this snippet of code and see if you can figure out what it is trying to do.作为一个例子，看一下这段代码，看看你是否能弄清楚它要做什么。

pub fn functional_blur(input: &Matrix) -> Matrix {
    assert!(input.width >= 3);
    assert!(input.height >= 3);

    // Stash away the top and bottom rows so they can be
    // directly copied across later
    let mut rows = input.rows();
    let first_row = rows.next().unwrap();
    let last_row = rows.next_back().unwrap();

    let top_row = input.rows();
    let middle_row = input.rows().skip(1);
    let bottom_row = input.rows().skip(2);

    let blurred_elements = top_row
        .zip(middle_row)
        .zip(bottom_row)
        .flat_map(|((top, middle), bottom)| blur_rows(top, middle, bottom));

    let elements: Vec<f32> = first_row
        .iter()
        .copied()
        .chain(blurred_elements)
        .chain(last_row.iter().copied())
        .collect();

    Matrix::new_row_major(elements, input.width, input.height)
}

fn blur_rows<'a>(
    top_row: &'a [f32],
    middle_row: &'a [f32],
    bottom_row: &'a [f32],
) -> impl Iterator<Item = f32> + 'a {
    // stash away the left-most and right-most elements so they can be copied across directly.
    let &first = middle_row.first().unwrap();
    let &last = middle_row.last().unwrap();

    // Get the top, middle, and bottom row of our 3x3 sub-matrix so they can be
    // averaged.
    let top_window = top_row.windows(3);
    let middle_window = middle_row.windows(3);
    let bottom_window = bottom_row.windows(3);

    // slide the 3x3 window across our middle row so we can get the average
    // of everything except the left-most and right-most elements.
    let averages = top_window
        .zip(middle_window)
        .zip(bottom_window)
        .map(|((top, middle), bottom)| top.iter().chain(middle).chain(bottom).sum::<f32>() / 9.0);

    std::iter::once(first)
        .chain(averages)
        .chain(std::iter::once(last))
}

Initialize After Construction 构造后初始化

In many languages, it is normal to call an object’s constructor and initialize its fields afterward (either manually or by calling some init() method). However, this goes against Rust’s general convention of “make invalid states unrepresentable”.在许多语言中，调用对象的构造函数并在之后初始化其字段是很正常的（手动或通过调用一些 init() 方法）。然而，这违背了 Rust 的一般惯例，即 “使无效的状态无法表示” 。

Say you are writing an NLP application and have a dictionary containing all the possible words you can handle.假设你正在编写一个 NLP 应用程序，并且有一个包含所有可能的单词的字典，你可以处理。

This is one way you could create the dictionary:这是你可以创建字典的一种方式：

let mut dict = Dictionary::new();
// read the file and populate some internal HashMap or Vec
dict.load_from_file("./words.txt")?;

However, writing Dictionary this way means it now has two (hidden) states - empty and populated.然而，这样写 Dictionary 意味着它现在有两个（隐藏的）状态 —— 空和填充。

All downstream code that uses the Dictionary will assume it’s been populated already and write code accordingly. This may include doing things like indexing into the dictionary with dict["word"] which may panic if "word" isn’t there.所有使用 Dictionary 的下游代码都会假定它已经被填充，并相应地编写代码。这可能包括做一些事情，比如用 dict["word"] 索引到字典中，如果 "word" 不在那里，就会导致很多恐慌。

Now you’ve opened yourself up to a situation where passing an empty dictionary to code that expects a populated dictionary may trigger a panic.现在你已经使自己进入到这样一个局面：把一个空的 dictionary 传递给期望有一个填充的 dictionary 的代码，这就引入了可怕的可能性。

But that’s completely unnecessary.但这是完全没有必要的。

Just make sure the Dictionary is usable immediately after constructing it instead of populating it after the fact.只要确保 Dictionary 在构造后能立即使用，而不是在事后填充它。

let dict = Dictionary::from_file("./words.txt")?;

impl Dictionary {
  fn from_file(filename: impl AsRef<Path>) -> Result<Self, Error> {
    let text = std::fs::read_to_string(filename)?;
    let mut words = Vec::new();
    for line in text.lines() {
      words.push(line);
    }
    Ok(Dictionary { words })
  }
}

Internally the Dictionary::from_file() might create an empty Vec and populate it incrementally, but it won’t be stored in the Dictionary’s words field yet so there is no assumption that it is populated and useful.在 Dictionary::from_file() 内部，可能会创建一个空的 Vec 并逐步填充它，但它还不会被存储在 Dictionary 的 words 字段中，所以不存在它被填充和有用的假设。

How frequently you fall into this anti-pattern depends a lot on your background and coding style.你陷入这种反模式是怎么个频率，很大程度上取决于你的背景和编码风格。

Functional languages are often completely immutable so you’ll fall into the idiomatic pattern naturally. After all, it’s kinda hard to create a half-initialized thing and populate it later when you aren’t allowed to mutate anything.Functional languages 通常是完全不可变的，所以你很自然地就会去用（这个）地道的用法。毕竟，当你不被允许改变任何东西的时候，要先创建一个半初始化的东西、然后在以后再填充它，这种事儿还是挺困难的。

On the other hand, OO languages are much happier to let you initialize an object after it has been constructed, especially because object references can be null by default and they have no qualms about mutability… You could argue this contributes to why OO languages have a propensity for crashing due to an unexpected NullPointerException.另一方面， OO languages 更乐于让你在对象构造完成后再对其初始化，特别是在于对象引用可以默认为空、而且他们对可变性还没有任何顾虑 …… 你可以说这就是为什么 OO languages 会有这样一种倾向：一不小心就要意外地遇上 NullPointerException 然后崩溃。

Defensive Copies 防御性拷贝

To point out the obvious, a really nice property of immutable objects is that you can rely on them to never change. However, in languages like Python and Java, immutability isn’t transitive - i.e. if x is an immutable object, x.y isn’t guaranteed to be immutable unless it was explicitly defined that way.要指出的是，不可变对象的一个非常好的属性是，你可以依靠它们永不改变。然而，在像 Python 和 Java 这样的语言中，不可变不会传递 —— 也就是说，就算 x 是不可变对象，那也保证不论 x.y 也不可变，除非它真的被明确定义成不可变。

This means it’s possible to write code like this…这意味着我们有可能写出这样的代码...

class ImmutablePerson:
  def __init__(self, name: str, age: int, addresses: List[str]):
    self._name = name
    self._age = age
    self._addresses = addresses

  # read-only properties
  @property
  def name(self): return self._name
  @property
  def age(self): return self._age
  @property
  def addresses(self): return self._addresses

Then someone else comes along and accidentally messes up the address list as part of their normal code.然后人家来用，却还是不小心弄乱了地址列表，而这只是一小部分普通的代码。

  # Note: the post office's API only works with with uppercase letters so we
  # need to pre-process the address list
  for i, address in enumerate(addresses):

  client = PostOfficeClient()

person = ImmutablePerson("Joe Bloggs", 42, ["123 Fake Street"])

  f"Dear {person.name}, I Nigerian prince. Please help me moving my monies.",
  person.addresses
)

print(person.addresses) # ["123 FAKE STREET"]

While I admit the example is a bit contrived, it’s not uncommon for functions to modify the arguments they are given. Normally this is fine, but when your ImmutablePerson assumes its addresses field will never change, it’s annoying for some random piece of code on the other side of the project to modify it without you knowing.虽然我承认这个例子有点矫揉造作，但函数修改它们所给的参数的情况并不少见。通常这是好的，但是当你的 ImmutablePerson 假设它的 addresses 字段永远不会改变时，项目另一边的一些随机代码却能够在你不知道的情况下修改它，这是很令人讨厌的。

The typical solution to this is to preemptively copy the list so even if the caller tries to mutate its contents, they’ll be mutating a copy and not the original addresses field.典型的解决方案是，预先复制列表，从而即使调用者试图改变其内容，被改的也只是个副本，而非原来那个 addresses 字段。

class ImmutablePerson:
  ...

  @property
  def addresses(self): return self._addresses.copy()

In general, you’ll see defensive copies being used anywhere code wants to be sure that another piece of code won’t modify some shared object at an inopportune time.一般来说，你会看到防御性拷贝会被用在任何想要确保共享对象不会给别的代码不恰当修改了的地方。

Considering this is an article about Rust, you’ve probably guessed what the root cause of this is - a combination of aliasing and mutation.考虑到这是一篇关于 Rust 的文章，你可能已经猜到了这是什么根源 —— 别名和突变的组合。

You’ve also probably guessed why defensive copies aren’t really necessary when writing Rust code - lifetimes and the “shared immutable XOR single mutable” rule for references means it just isn’t possible for code to modify something without first asking its original owner for mutable access or explicitly opting into shared mutation by using a type like std::sync::Mutex<T>.你可能也猜到了为什么在编写 Rust 代码时，防御性拷贝并不是真正必要的 —— 生命期和引用的 “共享不可变 XOR 单一可变” 规则意味着，如果不首先向其原始所有者请求可变访问、或者通过使用 std::sync::Mutex<T> 这样的类型明确选择共享变异，代码就不可能修改某些东西。

You may sometimes see people using clone() to get around borrow checker errors, and exclaim “Ha! See, Rust forces you to make defensive copies too!”你有时会看到有人使用 clone() 来绕过借贷检查器的错误，并感叹道： “H哈！看， Rust 也强迫你做防御性拷贝！”

To which I would argue that these copies are mostly caused by a lack of familiarity with lifetimes, or an architecture issue which forces the programmer to make more copies than they need to.对此，我想说的是，这些拷贝大多是由于对生命期的不熟悉造成的，或者是一个架构问题，迫使程序员做了比他们需要的更多拷贝。

The Iterator Trait 迭代器特性

At its most simplest, an iterator is just something with a next() method that may return some Item or None.在最简单的情况下，迭代器只是一个具有 next() 方法的东西，可以返回一些 Item 或 None 。

This is all encapsulated in the std::iter::Iterator trait, with the definition looking something like this:这一切都被封装在 std::iter::Iterator trait 中，其定义是这样的：

trait Iterator {
    type Item;

    fn next(&mut self) -> Option<Self::Item>;
}

Now you may be wondering why most Rust code doesn’t have calls to next() scattered throughout, and that’s where Syntactic Sugar comes in. This is where the compiler will see certain high level constructs in your code and transform them into a lower level form.现在你可能想知道为什么大多数 Rust 代码中没有散布对 next() 的调用，这就是语法糖（ Syntactic Sugar ）的作用。在这里，编译器会看到你的代码中的某些高级结构，并将它们转化为较低级别的形式。

The humble for-loop is probably the most familiar instance of this syntactic sugar.简陋的 for-loop 可能是这种句法糖最熟悉的例子。

For example, when the compiler sees this…例如，当编译器看到这个 ...

for item in 0..5 {
    println!("{}", item);
}

… the code will be transformed into this…... 代码就会被转化为这个 ...

let mut iterator = (0..5).into_iter();

while let Some(item) = iterator.next() {
    println!("{}", item);
}

What this does is turn the range expression, 0..5, into an iterator and then keep calling its next() method until we get None.这所做的是把范围表达式 0..5 变成一个迭代器，然后不断调用它的 next() 方法，直到我们得到 None 。

This into_iter() method (backed by the std::iter::IntoIterator trait) is a way to transform something into an iterator.这个 into_iter() 方法（由 std::iter::IntoIterator 特性支持）是一种将某物转化为迭代器的方法。

Some common types that implement IntoIterator are Vec<T>, Option<T>, and most importantly, anything implementing the Iterator trait (which will just return itself).一些实现 IntoIterator 的常见类型是 Vec<T> 、 Option<T> ，以及最重要的，任何实现 Iterator trait 的东西 （它将只是返回自己）。

Fibonacci 斐波那契

Now we have some basic familiarity with iterators, let’s create our own iterator over all the Fibonacci numbers that fit into a u32.现在我们对迭代器有了一些基本的了解，让我们在所有适合 u32 的斐波那契数上创建我们自己的迭代器。

First we’ll create a new Fibonacci type.首先，我们将创建一个新的 Fibonacci （斐波那契）类型。

struct Fibonacci {
    a: u32,
    b: u32,
}

Then we implement the Iterator trait. The idea is that each call to next() will do just enough work to generate the next number in the sequence.然后我们实现 Iterator 的特性。我们的想法是，每次调用 next() 将做足够的工作来生成序列中的下一个数字。

impl Iterator for Fibonacci {
    type Item = u32;

    fn next(&mut self) -> Option<u32> {
        // Try to add "a" and "b", returning None if the addition would
        // overflow to signal that we've reached the end of the sequence.
        let next_term = self.a.checked_add(self.b)?;

        self.a = self.b;
        self.b = next_term;

        Some(next_term)
    }
}

You’ll notice we use the checked_add() method here instead of the normal + operator. This is important because we are working with fixed-size integers and we need a stopping condition.你会注意到我们在这里使用了 checked_add() 方法，而不是普通的 + 运算符。这很重要，因为我们是在处理固定大小的整数，我们需要一个停止条件。

Without it, we would reach 2_971_215_073 and either panic (in debug mode) overflow (in release mode). This is Rust’s way of telling us that we if we are creating bigger and bigger numbers we should handle the situation when our numbers get too big for the integer type we are using.如果没有它，我们会在达到 2_971_215_073 时出现恐慌（在调试模式）和溢出（在发布模式）。这是 Rust 告诉我们的方式，如果我们正在创建越来越大的数字，我们应该处理当我们的数字对于我们使用的整数类型来说太大的情况。

}

    }
}

First we need a struct which keeps track of the slice and the index of the next item to read.首先，我们需要一个结构来跟踪切片和要读取的下一个项目的索引。

struct SliceIter<'a, T> {
    slice: &'a [T],
    index: usize,
}

Our implementation then just uses the slice’s get() method to get the item at that index or None if the index is past the end of the slice. We then just use ? to return None if there was no item, then increment our index before yielding the item to the caller.然后，我们的实现只是使用 切片的 get() 方法 来获取该索引下的项目，如果该索引已经超过了切片的末尾，则返回 None 。然后，如果没有项目，我们就使用 ? 来返回 None ，然后在把项目交给调用者之前增加我们的索引。

impl<'a, T> Iterator for SliceIter<'a, T> {

        Some(item)
    }
}
