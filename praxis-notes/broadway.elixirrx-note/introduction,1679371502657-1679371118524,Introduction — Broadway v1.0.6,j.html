Broadway is a library for building concurrent and multi-stage data ingestion and data processing pipelines with Elixir. Broadway pipelines are concurrent and robust, thanks to the Erlang VM and its actors. It features:Broadway 是一个用于使用 Elixir 构建并发和多阶段数据摄取和数据处理管道的库。得益于 Erlang VM 及其参与者，Broadway 管道是并发且健壮的。它的特点：

  • Back-pressure背压
  • Automatic acknowledgements at the end of the pipeline管道末端的自动确认
  • Batching配料
  • Fault tolerance容错性
  • Graceful shutdown优雅关机
  • Built-in testing内置测试
  • Custom failure handling自定义故障处理
  • Ordering and partitioning排序和分区
  • Rate-limiting限速
  • Metrics指标

Amazon Simple Queue Service (SQS) is a highly scalable distributed message queuing service provided by Amazon.com. AWS SQS offers two types of message queues:Amazon Simple Queue Service (SQS) 是 Amazon.com 提供的高度可扩展的分布式消息队列服务。 AWS SQS 提供两种类型的消息队列：

  • Standard标准

    • Nearly unlimited throughput几乎无限的吞吐量
    • Best-effort ordering尽力而为排序
    • At-least-once delivery至少一次交付

  • FIFO先进先出

    • Limited number of transactions per second (TPS). See Amazon SQS FIFO developer guide for more information on limits.每秒事务数 (TPS) 有限。有关限制的更多信息，请参阅 Amazon SQS FIFO 开发人员指南。
    • Order in which messages are sent/received is strictly preserved严格保留发送/接收消息的顺序
    • Exactly-once delivery恰好一次交付

Broadway can work seamlessly with both, Standard and FIFO queues.Broadway 可以与标准队列和 FIFO 队列无缝协作。

define-the-pipeline-configuration

 Define the pipeline configuration 定义管道配置 定义管道配置

Broadway is a process-based behaviour and to define a Broadway pipeline, we need to define three functions: start_link/1, handle_message/3 and handle_batch/4. We will cover start_link/1 in this section and the handle_ callbacks in the next one.Broadway 是一种基于流程的行为，要定义 Broadway 管道，我们需要定义三个函数：start_link/1、handle_message/3 和 handle_batch/4。我们将在本节中介绍 start_link/1，在下一节中介绍 handle_ 回调。

Similar to other process-based behaviour, start_link/1 simply delegates to Broadway.start_link/2, which should define the producers, processors, and batchers in the Broadway pipeline. Assuming we want to consume messages from a queue called my_queue, the minimal configuration would be:与其他基于流程的行为类似，start_link/1 只是委托给 Broadway.start_link/2，它应该定义 Broadway 管道中的生产者、处理者和批处理者。假设我们要使用来自名为 my_queue 的队列的消息，最小配置为：

defmodule MyBroadway do
  use Broadway

  alias Broadway.Message

  def start_link(_opts) do
    Broadway.start_link(__MODULE__,
      name: __MODULE__,
      producer: [
        module: {BroadwaySQS.Producer,
                 queue_url: "https://us-east-2.queue.amazonaws.com/100000000001/my_queue"}
      ],
      processors: [
        default: []
      ],
      batchers: [
        default: [
          batch_size: 10,
          batch_timeout: 2000
        ]
      ]
    )
  end

  ...callbacks...
end

The above configuration also assumes that you have the AWS credentials set up in your environment, for instance, by having the AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY environment variables set. If that's not the case, you will need to pass that information to the client so it can properly connect to the AWS servers. Here is how you can do it:上述配置还假定您已在您的环境中设置 AWS 凭证，例如，通过设置 AWS_ACCESS_KEY_ID 和 AWS_SECRET_ACCESS_KEY 环境变量。如果不是这种情况，您将需要将该信息传递给客户端，以便它可以正确连接到 AWS 服务器。以下是您的操作方法：

...
producer: [
  module:
    {BroadwaySQS.Producer,
     queue_url: "https://us-east-2.queue.amazonaws.com/100000000001/my_queue",
     config: [
       access_key_id: "YOUR_AWS_ACCESS_KEY_ID",
       secret_access_key: "YOUR_AWS_SECRET_ACCESS_KEY"
     ]}
]
...

For a full list of options for BroadwaySQS.Producer, please see BroadwaySQS documentation.有关 BroadwaySQS.Producer 选项的完整列表，请参阅 BroadwaySQS 文档。

For general information about setting up Broadway, see Broadway module docs as well as Broadway.start_link/2.有关设置 Broadway 的一般信息，请参阅 Broadway 模块文档以及 Broadway.start_link/2。

Note: Even though batching is optional since Broadway v0.2, we recommend that all Amazon SQS pipelines have at least a default batcher. This lets you control the exact batch size and frequency that messages are acknowledged to Amazon SQS, often leading to pipelines that are more cost and time efficient.注意：尽管从 Broadway v0.2 开始批处理是可选的，但我们建议所有 Amazon SQS 管道至少有一个默认批处理程序。这让您可以控制向 Amazon SQS 确认消息的确切批量大小和频率，通常会导致管道成本和时间效率更高。

implement-broadway-callbacks

 Implement Broadway callbacks 实施百老汇回调 实施百老汇回调

In order to process incoming messages, we need to implement the required callbacks. For the sake of simplicity, we're considering that all messages received from the queue are just numbers:为了处理传入的消息，我们需要实现所需的回调。为了简单起见，我们考虑从队列接收的所有消息都只是数字：

defmodule MyBroadway do
  use Broadway

  alias Broadway.Message

  ...start_link...

  @impl true
  def handle_message(_, %Message{data: data} = message, _) do
    message
    |> Message.update_data(fn data -> data * data end)
  end

  @impl true
  def handle_batch(_, messages, _, _) do
    list = messages |> Enum.map(fn e -> e.data end)
    IO.inspect(list, label: "Got batch of finished jobs from processors, sending ACKs to SQS as a batch.")
    messages
  end
end

We are not doing anything fancy here, but it should be enough for our purpose. First we update the message's data individually inside handle_message/3 and then we print each batch inside handle_batch/4.我们在这里没有做任何花哨的事情，但对于我们的目的来说应该足够了。首先，我们在 handle_message/3 中单独更新消息的数据，然后在 handle_batch/4 中打印每个批次。

For more information, see Broadway.handle_message/3 and Broadway.handle_batch/4.有关详细信息，请参阅 Broadway.handle_message/3 和 Broadway.handle_batch/4。

run-the-broadway-pipeline

 Run the Broadway pipeline 运行百老汇管道 运行百老汇管道

To run your Broadway pipeline, you just need to add as a child in a supervision tree. Most applications have a supervision tree defined at lib/my_app/application.ex. You can add Broadway as a child to a supervisor as follows:要运行 Broadway 流水线，您只需将其添加为监督树中的孩子。大多数应用程序都在 lib/my_app/application.ex 中定义了监督树。您可以将 Broadway 作为子级添加到主管，如下所示：

children = [
  {MyBroadway, []}
]

Supervisor.start_link(children, strategy: :one_for_one)

Now the Broadway pipeline should be started when your application starts. Also, if your Broadway has any dependency (for example, it needs to talk to the database), make sure that Broadway is listed after its dependencies in the supervision tree.现在 Broadway 管道应该在您的应用程序启动时启动。此外，如果您的 Broadway 有任何依赖项（例如，它需要与数据库对话），请确保 Broadway 在监督树中的依赖项之后列出。

tuning-the-configuration

 Tuning the configuration 调整配置 调整配置

Some of the configuration options available for Broadway come already with a "reasonable" default value. However those values might not suit your requirements. Depending on the number of messages you get, how much processing they need and how much IO work is going to take place, you might need completely different values to optimize the flow of your pipeline. The concurrency option available for every set of producers, processors and batchers, among with max_demand, batch_size, and batch_timeout can give you a great deal of flexibility.Broadway 可用的一些配置选项已经带有“合理”的默认值。但是，这些值可能不符合您的要求。根据您收到的消息数量、它们需要多少处理以及将要进行多少 IO 工作，您可能需要完全不同的值来优化您的管道流。可用于每组生产者、处理器和批处理程序的并发选项，其中包括 max_demand、batch_size 和 batch_timeout，可以为您提供很大的灵活性。

The concurrency option controls the concurrency level in each layer of the pipeline. See the notes on Producer concurrency and Batcher concurrency for details.并发选项控制管道每一层的并发级别。详见Producer并发和Batcher并发的注释。

Here's an example on how you could tune them according to your needs.这是一个关于如何根据需要调整它们的示例。

defmodule MyBroadway do
  use Broadway

  def start_link(_opts) do
    Broadway.start_link(__MODULE__,
      name: __MODULE__,
      producer: [
        ...
        concurrency: 10,
      ],
      processors: [
        default: [
          concurrency: 100,
          max_demand: 1,
        ]
      ],
      batchers: [
        default: [
          batch_size: 10,
          concurrency: 10,
        ]
      ]
    )
  end

  ...callbacks...
end

In order to get a good set of configurations for your pipeline, it's important to respect the limitations of the servers you're running, as well as the limitations of the services you're providing/consuming data to/from. Broadway comes with telemetry, so you can measure your pipeline and help ensure your changes are effective.为了为您的管道获得一组良好的配置，重要的是要尊重您正在运行的服务器的限制，以及您向/从中提供/使用数据的服务的限制。 Broadway 带有遥测功能，因此您可以测量您的管道并帮助确保您的更改有效。

tuning-the-configuration

Some of the configuration options available for Broadway come already with a "reasonable" default value. However, those values might not suit your requirements. Depending on the number of records you get, how much processing they need and how much IO work is going to take place, you might need completely different values to optimize the flow of your pipeline. The concurrency option available for every set of producers, processors and batchers, along with batch_size and batch_timeout can give you a great deal of flexibility. By setting the concurrency option, you define the number of concurrent processes that will be started by Broadway, allowing you to have full control over the concurrency level in each layer of the pipeline. Keep in mind that since the concurrency model provided by Kafka is based on partitioning, in order to take full advantage of this model, you need to set the concurrency option for your processors and batchers accordingly. Having less concurrency than topic/partitions assigned will result in individual processors handling more than one partition, decreasing the overall level of concurrency. Therefore, if you want to always be able to process messages at maximum concurrency (assuming you have enough resources to do it), you should increase the concurrency up front to make sure you have enough processors to handle the extra records received from new partitions assigned.通过设置并发选项，您可以定义 Broadway 将启动的并发进程数，从而使您可以完全控制管道每一层的并发级别。请记住，由于 Kafka 提供的并发模型基于分区，为了充分利用此模型，您需要相应地为处理器和批处理程序设置并发选项。并发性低于分配的主题/分区将导致单个处理器处理多个分区，从而降低总体并发水平。因此，如果您希望始终能够以最大并发度处理消息（假设您有足够的资源来做到这一点），您应该预先增加并发度以确保您有足够的处理器来处理从分配的新分区收到的额外记录.

There are other options that you may want to take a closer look when tuning your configuration. The :max_bytes option, for instance, belongs to the :fetch_config group and defines the maximum amount of data to be fetched at a time from a single partition. The default is 1048576 (1 MiB). For more information and other fetch options, please refer to the "Fetch config options" in the official BroadwayKafka documentation.在调整配置时，您可能还需要仔细查看其他选项。例如，:max_bytes 选项属于 :fetch_config 组，它定义了一次从单个分区中获取的最大数据量。默认值为 1048576 (1 MiB)。设置更大的值可以以更多的内存消耗为代价提高吞吐量。有关更多信息和其他获取选项，请参阅 BroadwayKafka 官方文档中的“获取配置选项”。

Other two important options are :offset_commit_interval_seconds and :offset_commit_on_ack. Both belong to the main configuration and they can make a huge impact on performance.其他两个重要选项是：offset_commit_interval_seconds 和：offset_commit_on_ack。两者都属于主要配置，它们会对性能产生巨大影响。

The :offset_commit_interval_seconds defines the time interval between two OffsetCommitRequest messages. The default is 5s.:offset_commit_interval_seconds 定义了两个 OffsetCommitRequest 消息之间的时间间隔。默认为 5 秒。

The :offset_commit_on_ack, when set to true, tells Broadway to send an OffsetCommitRequest immediately after each acknowledgement, bypassing any interval defined in :offset_commit_interval_seconds. Setting this option to false can increase performance since any commit requests will start respecting the :offset_commit_interval_seconds option. This will usually result in fewer requests to be sent to the server. However, setting long commit intervals might lead to a large number of duplicated records to be processed after a server restart or connection loss. Since it is always possible that duplicate messages will be received by consumers, make sure your logic is idempotent when consuming records to avoid inconsistencies. Also, bear in mind that the negative performance impact might be insignificant if you're using batchers since only one commit request will be performed per batch. As a basic rule, always take into account the values of batch_size and batch_timeout whenever you're tuning :offset_commit_interval_seconds and :offset_commit_on_ack.:offset_commit_on_ack 设置为 true 时，告诉 Broadway 在每次确认后立即发送 OffsetCommitRequest，绕过 :offset_commit_interval_seconds 中定义的任何间隔。将此选项设置为 false 可以提高性能，因为任何提交请求都将开始遵守 :offset_commit_interval_seconds 选项。这通常会导致发送到服务器的请求更少。但是，设置较长的提交间隔可能会导致在服务器重新启动或连接丢失后处理大量重复的记录。由于消费者总是有可能收到重复的消息，因此在使用记录时确保您的逻辑是幂等的，以避免不一致。另外，请记住，如果您使用批处理程序，负面性能影响可能微不足道，因为每批只执行一个提交请求。作为一个基本规则，无论何时调整 :offset_commit_interval_seconds 和 :offset_commit_on_ack，都要考虑 batch_size 和 batch_timeout 的值。

handling-failed-messages

 Handling failed messages 处理失败的消息 处理失败消息

broadway_kafka never stops the flow of the stream, i.e. it will always ack the messages even when they fail. Unlike queue-based connectors, where you can mark a single message as failed. In Kafka that's not possible due to its single offset per topic/partition ack strategy. If you want to reprocess failed messages, you need to roll your own strategy. A possible way to do that is to implement handle_failed/2 and send failed messages to a separated stream or queue for later processing.broadway_kafka 永远不会停止流的流动，即即使消息失败，它也会始终确认消息。与基于队列的连接器不同，您可以在其中将单个消息标记为失败。在 Kafka 中，由于每个主题/分区确认策略的单个偏移量，这是不可能的。如果你想重新处理失败的消息，你需要推出自己的策略。一种可能的方法是实现 handle_failed/2 并将失败的消息发送到单独的流或队列以供以后处理。

defmodule Counter do

  def start_link(number) do
    GenStage.start_link(Counter, number)
  end

  def init(counter) do
    {:producer, counter}
  end

  def handle_demand(demand, counter) when demand > 0 do
    events = Enum.to_list(counter..counter+demand-1)
    {:noreply, events, counter + demand}
  end
end

  use Broadway

  alias Broadway.Message

  def start_link(_opts) do
    Broadway.start_link(__MODULE__,
      name: __MODULE__,
      producer: [
        module: {Counter, 1},
        transformer: {__MODULE__, :transform, []}
      ],
      processors: [
        default: [concurrency: 10]
      ],
      batchers: [
        default: [concurrency: 2, batch_size: 5],
      ]
    )
  end

  ...callbacks...

  def transform(event, _opts) do
    %Message{
      data: event,
      acknowledger: {__MODULE__, :ack_id, :ack_data}
    }
  end

  def ack(:ack_id, successful, failed) do
    # Write ack code here
  end
end

The only exception are the producers, which contain external code and are expected to fail. Broadway automatically handles those failures by making processors automatically resubscribe to producers in case of crashes.唯一的例外是生产者，它包含外部代码并且预计会失败。如果生产者崩溃，它将由其监管者重新启动，而不会出现级联故障，直到达到其最大重启次数。百老汇通过让处理器在发生崩溃时自动重新订阅生产者来自动处理这些故障。

graceful-shutdowns

 Graceful shutdowns 优雅关机 优雅关机

The cascading failures aspect also provides safe semantics for graceful shutdown. We know that either all processes are running OR they are all being shutdown. Therefore, to gracefully shutdown the supervision tree, a terminator process is activated, which starts the following steps:级联故障方面还为正常关闭提供安全语义。我们知道要么所有进程都在运行，要么它们都被关闭。因此，为了优雅地关闭监督树，一个终止进程被激活，它开始以下步骤：

  1. It notifies the first layer of processors that they should not resubscribe to producers once they exit它通知第一层处理器一旦退出就不要重新订阅生产者

  2. It tells all producers to no longer accept demand, flush all current events, and then shutdown它告诉所有生产者不再接受需求，刷新所有当前事件，然后关闭

  3. It then monitors and waits for a confirmation message from batch processors. At this point, the terminator is effectively blocking the supervisor until all events have been processed然后它监视并等待来自批处理器的确认消息。在这一点上，终结者有效地阻塞了监督者，直到所有事件都被处理完

This triggers a cascade effect where processors notice all of its producers have been cancelled, causing them to flush their own events and cancels the stages downstream, and so on and so on. This happens until batch processors notice all of their producers have been cancelled, effectively notifying the terminator to shutdown, allowing the outer most supervisor to go on and fully terminate all stages, which at this point have flushed all events.这会触发级联效应，处理器会注意到其所有生产者都已被取消，从而导致他们刷新自己的事件并取消下游阶段，依此类推。这种情况会发生，直到批处理器注意到他们的所有生产者都被取消，有效地通知终结者关闭，允许最外面的监管者继续并完全终止所有阶段，此时已经刷新了所有事件。
