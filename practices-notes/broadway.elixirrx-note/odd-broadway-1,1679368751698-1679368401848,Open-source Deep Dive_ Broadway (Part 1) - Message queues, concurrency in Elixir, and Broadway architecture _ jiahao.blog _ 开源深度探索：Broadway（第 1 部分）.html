What are message queues?什么是消息队列？

Note! While Broadway can integrate with many types of data sources, the core examples given in the project focus on message queues as the primary data source.笔记！虽然 Broadway 可以与多种类型的数据源集成，但项目中给出的核心示例侧重于将消息队列作为主要数据源。

Message queues are like containers that hold sequences of work objects — called messages — that are to be consumed and processed. It aids with building asynchronous modular and concurrent systems.消息队列就像容器，其中包含要使用和处理的工作对象序列（称为消息）。它有助于构建异步模块化和并发系统。

Messages are created and delivered to these queues by producers and taken from these queues for processing by ** consumers.** These messages can vary from something as simple as plain information to more complex structures like requests or — in our case — transaction information.消息由生产者创建并传递到这些队列，并从这些队列中取出以供 ** 消费者处理。** 这些消息可以从简单的纯信息到更复杂的结构，如请求或 - 在我们的例子中 - 事务信息。

Message queues are useful for decentralising the communication mechanism of large systems by acting as a medium for exchanging events between systems which allows for systems to be easily scaled and distributed.消息队列可用作系统之间交换事件的媒介，从而使系统易于扩展和分布，从而有助于分散大型系统的通信机制。

This is a reduced explanation of what a message queue is and what it is capable of. For more information about message queues, the Amazon documentation and this blog post by CloudAMQP are good places to start.这是对什么是消息队列及其功能的简化解释。有关消息队列的更多信息，Amazon 文档和 CloudAMQP 的这篇博文是不错的起点。

Concurrency in ElixirElixir 中的并发

Broadway relies heavily on concurrency in Elixir. The topology (architecture) of a pipeline is built on top of processes and many of the features are achieved using the robust concurrency model of Elixir. So what exactly is the concurrency model in Elixir?Broadway 严重依赖 Elixir 中的并发性。管道的拓扑（架构）建立在进程之上，许多功能是使用 Elixir 的强大并发模型实现的。那么 Elixir 中的并发模型到底是什么？

Elixir employs the actor concurrency model. In this model, actors are defined as self-isolated units of processing. In Elixir, these actors are called processes and they are managed by the Erlang VM . Elixir code is run in each process and a default/main process is akin to that of the main thread in other concurrency models.Elixir 采用 actor 并发模型。在这个模型中，参与者被定义为自我隔离的处理单元。在 Elixir 中，这些参与者被称为进程，它们由 Erlang VM 管理。 Elixir 代码在每个进程中运行，默认/主进程类似于其他并发模型中的主线程。

Each process communicates via asynchronous message passing. Think of a process as a mailbox of sorts; it has a “bin” to receive incoming messages and it possess an “address” for other processes to identify it by.每个进程通过异步消息传递进行通信。将进程想象成某种邮箱；它有一个“bin”来接收传入的消息，并且它拥有一个“地址”供其他进程识别它。

The unique aspect of this model is the lack of shared mutable state that other concurrency models rely on. Rather, state is exclusive to each process.该模型的独特之处在于缺少其他并发模型所依赖的共享可变状态。相反，状态是每个进程独有的。

In order for the state of a process to be altered, the owner process must make the alteration either on request or internally due to certain changes.为了更改流程的状态，所有者流程必须根据请求或由于某些更改而在内部进行更改。

The topic of concurrency in Elixir is vast and Elixir provides many other features surrounding its concurrency model such as GenServer. This section is a short preview of what the actor concurrency model and concurrency in Elixir is all about. For more information, you can refer to this thesis paper and the Wikipedia article talking about the actor concurrency model and the official documentation and this tutorial on OTP in Elixir for more examples of concurrency in Elixir.Elixir 中的并发主题非常广泛，Elixir 围绕其并发模型提供了许多其他功能，例如 GenServer。本节简要介绍 Actor 并发模型和 Elixir 中的并发性。有关更多信息，您可以参考这篇论文和 Wikipedia 文章谈论 actor 并发模型和官方文档以及关于 Elixir 中 OTP 的教程以获取更多 Elixir 并发示例。

Cue the producer/consumer model提示生产者/消费者模型

Using the actor concurrency model as a foundation, another concurrency pattern can be modelled in Elixir — the producer/consumer model.使用 actor 并发模型作为基础，可以在 Elixir 中建模另一种并发模式——生产者/消费者模型。

This model aims to allow for decoupled data production and consumption by setting up two separate processes to handle each task — effectively creating a logical separation of concerns.该模型旨在通过设置两个单独的流程来处理每个任务来实现数据生产和消费的分离——有效地创建关注点的逻辑分离。

However, the producer/consumer model faces a critical issue — what happens if the producer generates excessive messages for the consumer? The consumer will be overwhelmed and will eventually fail trying to keep up with processing that many messages. This is where back pressure comes into play.然而，生产者/消费者模型面临着一个关键问题——如果生产者为消费者生成过多的消息会怎样？消费者将不堪重负，最终将无法跟上处理那么多消息的步伐。这就是背压发挥作用的地方。

Back pressure is a control mechanism for how much a producer should emit based on consumer demand, consumer message buffering, or limited sampling背压是一种控制机制，用于控制生产者应根据消费者需求、消费者消息缓冲或有限采样发出多少

Back pressure avoids the problem of overloading the consumer with messages by applying one of or a combination of the three methods mentioned above (more information in the link here).背压通过应用上述三种方法中的一种或组合来避免消息使消费者过载的问题（更多信息在此处的链接中）。

The next frontier (of concurrency): GenStage下一个（并发）前沿：GenStage

Seeing the value of having a standard implementation for the producer/consumer model, the Elixir team decided to develop exactly that.看到生产者/消费者模型的标准实现的价值，Elixir 团队决定开发这种模型。

GenStage is a specification for exchanging events between producers and consumers with back pressure between Elixir processesGenStage 是一种在生产者和消费者之间交换事件的规范，在 Elixir 进程之间有背压

Producers emit events to consumers for processing. The events can be of any structure.生产者将事件发送给消费者进行处理。事件可以是任何结构。

The control mechanism used is a demand system. Consumers inform producers of how many events they can handle (demand) and producers emits no more than the demanded amount. This ensures that the consumers are capable of handling the events emitted.使用的控制机制是需求系统。消费者通知生产者他们可以处理多少事件（需求），生产者发出的事件不超过要求的数量。这确保消费者能够处理发出的事件。

Producer-consumers behave like both producers and consumers. They are used to perform transformations on events emitted by the producer before they are emitted to the consumer.生产者-消费者的行为既像生产者又像消费者。它们用于在生产者发出的事件发送给消费者之前对其执行转换。

Similar to GenServer, stages in GenStage exchange events through callbacks.与 GenServer 类似，GenStage 中的阶段通过回调交换事件。

When a demand is handled — i.e. producer emits events and demanding consumer handles these events — another demand is made, creating a cycle where both stages are always working - ideally.当一个需求被处理时——即生产者发出事件，要求消费者处理这些事件——另一个需求被提出，创造一个循环，两个阶段总是工作——理想情况下。

GenStage is a powerful tool in an Elixir developer’s arsenal. More information can be found in the official announcement where a little bit of history of how GenStage came to be was discussed and in a talk by José Valim — creator of Elixir.GenStage 是 Elixir 开发人员库中的一个强大工具。更多信息可以在官方公告中找到，其中讨论了一些关于 GenStage 如何诞生的历史，以及 Elixir 的创始人何塞·瓦利姆 (José Valim) 的演讲。

Pipeline architecture流水线架构

It is at this juncture where it would be important to clarify the term “producer”. In both message queues and GenStage, a producer is a creator of messages or events. However, in Broadway, a producer is both a consumer of messages and an emitter of events.正是在这个时刻，澄清“生产者”一词很重要。在消息队列和 GenStage 中，生产者是消息或事件的创建者。然而，在百老汇，生产者既是消息的消费者又是事件的发射者。

For the rest of the article, the following definitions for the following terminology will be used:对于本文的其余部分，将使用以下术语的以下定义：

How it’s made: Producers制作方法：制作人

Producers are built using a pattern similar to the strategy pattern but modified to integrate with the concurrency system in Elixir.生产者是使用类似于策略模式的模式构建的，但经过修改以与 Elixir 中的并发系统集成。

Different data sources require different methods of establishing connections and receiving messages. Thus, we break up the producer process into two modules — ProducerStage defines the behavior for enforcing the rate limit while a dynamically loaded module defines the behavior for establishing a connection to the data source and receiving messages.不同的数据源需要不同的建立连接和接收消息的方法。因此，我们将生产者进程分解为两个模块——ProducerStage 定义了强制执行速率限制的行为，而动态加载的模块定义了与数据源建立连接和接收消息的行为。

ProducerStage assumes that the dynamically loaded module contains the typical GenStage callbacks like handle_call and handle_demand and uses them for things like rate limiting.ProducerStage 假定动态加载的模块包含典型的 GenStage 回调，如 handle_call 和 handle_demand，并将它们用于诸如速率限制之类的事情。

The ProducerStage behaves as the context while the dynamic module behaves as the strategy. The dynamic module adopts the Producer module — which defines two callbacks for managing the overall producer life-cycle.ProducerStage 充当上下文，而动态模块充当策略。动态模块采用生产者模块——它定义了两个回调来管理整个生产者生命周期。

To load the module dynamically, the module name is passed to ProducerStage as an argument. To keep the producer as a single process, we call the init function of the module directly when initialising the ProducerStage. This way, the module will initialise under the newly spawned process for ProducerStage rather than spawning an entirely new process.要动态加载模块，模块名称作为参数传递给 ProducerStage。为了使生产者保持为单个进程，我们在初始化 ProducerStage 时直接调用模块的 init 函数。这样，该模块将在新生成的 ProducerStage 进程下初始化，而不是生成一个全新的进程。

How it’s made: Processors它是如何制造的：处理器

Processors are created using a concept similar to inheritance in object-oriented programming. This idea comes from the need to standardise the subscription logic of producer-consumers and consumers.处理器是使用类似于面向对象编程中的继承的概念创建的。这个想法源于需要规范生产者-消费者和消费者的订阅逻辑。

Rate limiting速率限制

Rate limiting refers to the act of limiting the amount of data that can be requested or processed in a given period of time速率限制是指限制在给定时间段内可以请求或处理的数据量的行为

Rate limiting is applied across producers within a single pipeline to control the number of events emitted within a given period of time.速率限制适用于单个管道内的生产者，以控制给定时间段内发出的事件数量。

This is especially useful when the hardware of the machine running the pipeline is not able to keep up with processing large numbers of events demanded at a time — possibly due to a poorly configured pipeline.当运行管道的机器的硬件无法跟上处理一次所需的大量事件时，这尤其有用——可能是由于管道配置不当。

Some producers do not leverage the rate limiting feature of Broadway. For instance, the RabbitMQ producer creates an active listener, which means that event emission is not inhibited by the rate limiter. Instead, events are emitted the moment messages are published to the message queue ( unless otherwise configured) .一些制作人不利用百老汇的速率限制功能。例如，RabbitMQ 生产者创建了一个主动监听器，这意味着事件发射不会被速率限制器抑制。相反，事件会在消息发布到消息队列时发出（除非另外配置）。

But for the producers that do leverage the rate limiting — such as the Amazon SQS producer — rate limiting is applied in two instances:但对于确实利用速率限制的生产者（例如 Amazon SQS 生产者），速率限制适用于两种情况：

  1. When consumers make demands to the producer-consumer or producer当消费者向生产者消费者或生产者提出要求时

    If the producer can still emit events, any demand made by the consumer will be handled by the producer. We take into account the rate limit threshold. If there are too many events to emit, the excess messages are stored in a message buffer that will have to be cleared later on.如果生产者仍然可以发出事件，则消费者提出的任何需求都将由生产者处理。我们考虑了速率限制阈值。如果要发出的事件太多，多余的消息将存储在消息缓冲区中，稍后必须清除该缓冲区。

    Each message that can be emitted will be transformed into the standard event structure that Broadway uses.可以发出的每条消息都将转换为百老汇使用的标准事件结构。

    If the producer can no longer emit messages, any demand made is stored in a demand buffer that is cleared later on.如果生产者不能再发出消息，则发出的任何需求都会存储在稍后清除的需求缓冲区中。

  2. When the rate limit is being reset after the given interval在给定时间间隔后重置速率限制时

    After the given interval, the rate limit threshold can be reset. However, we may have accumulated demands and messages in their respective buffers. We may find that the threshold has not been met before we reset it. Thus, we can use this remaining threshold to clear any lingering demands and messages stored in their respective buffers.在给定的时间间隔后，可以重置速率限制阈值。但是，我们可能在各自的缓冲区中积累了需求和消息。我们可能会发现在我们重置它之前还没有达到阈值。因此，我们可以使用这个剩余阈值来清除存储在各自缓冲区中的任何挥之不去的需求和消息。

    Once we have cleared as many messages as our remaining threshold allows, we will reset the threshold and schedule for another reset. These resets are scheduled at fixed intervals.一旦我们清除了剩余阈值允许的尽可能多的消息，我们将重置阈值并安排另一次重置。这些重置以固定的时间间隔进行安排。

The rate limiting threshold is maintained as an atomic (discussed later on). This atomic array is generated by the RateLimiter process. This module handles all behavior surrounding working with the rate limit threshold. ProducerStage handles the actual logic of managing the demands of consumers.速率限制阈值保持为原子（稍后讨论）。这个原子数组是由 RateLimiter 进程生成的。该模块处理与速率限制阈值相关的所有行为。 ProducerStage 处理管理消费者需求的实际逻辑。

When the producer cannot emit any more events, i.e. the threshold has been reached, an internal state is set to :closed to avoid future demands from being handled.当生产者不能再发出任何事件时，即达到阈值时，内部状态将设置为 :closed 以避免处理未来的需求。

Batching配料

Batching groups events based on given properties and sends them to designated “sub-pipelines” or batch processors to be handled. For instance, we might design a pipeline that stores events with even numbers in an S3 bucket and ones with odd numbers on Google Drive.批处理根据给定的属性对事件进行分组，并将它们发送到指定的“子管道”或批处理器进行处理。例如，我们可能会设计一个管道，将偶数事件存储在 S3 存储桶中，将奇数事件存储在 Google Drive 中。

The Batcher process is assigned unique names for identification and events that are emitted from the producer must be tagged to a batcher. Failure to do so will result in a runtime error. This only applies if batching is enabled.Batcher 进程被分配了唯一的名称用于标识，并且从生产者发出的事件必须标记为 batcher。不这样做将导致运行时错误。这仅适用于启用批处理的情况。

In order for the producer to send the appropriate events to the respective batcher, a PartitionDispatcher is used. Essentially, it defines the behavior of how events are emitted to consumers. A PartitionDispatcher dispatches events to certain consumers based on a given criteria (defined as a hash function). In this case, the criterion is the name of the batcher from the given event. This means that when we assign a batcher to the event, it will be dispatched to only that batcher. More information about dispatchers in GenStage can be found in the official documentation.为了让生产者将适当的事件发送到相应的批处理程序，使用了 PartitionDispatcher。本质上，它定义了事件如何发送给消费者的行为。 PartitionDispatcher 根据给定的标准（定义为散列函数）将事件分派给某些消费者。在这种情况下，标准是来自给定事件的批处理程序的名称。这意味着当我们为事件分配一个批处理程序时，它只会被分派给那个批处理程序。有关 GenStage 中调度程序的更多信息，请参阅官方文档。

Even within the batcher, further grouping can be made based on a batch key assigned to the event. This may be used to ensure that certain events are processed together. Internally, the batcher will accumulate events before emitting them. However, as it cannot sit around accumulating events forever, a batch is emitted at regular intervals regardless of how many events are stored in it.即使在批处理程序中，也可以根据分配给事件的批处理键进行进一步分组。这可用于确保某些事件一起处理。在内部，批处理程序将在发出事件之前累积事件。但是，由于它不能永远累积事件，因此无论其中存储了多少事件，都会定期发出一个批处理。

The BatchProcessor process handles a single batch at a time. It is similar to a regular processor, except it works on a batch of events. The handle_batch callback is used here.BatchProcessor 进程一次处理一个批次。它类似于常规处理器，只是它处理一批事件。这里使用了 handle_batch 回调。

Built-in testing内置测试

To test the pipeline, we should focus on ensuring that the data processing aspect of the pipeline works as intended. However, as we rely on external services for input, it would be hard to coordinate a test suite to work with a live data source as we may not be able to replicate the data source or publish data to the data source at will due to access limitations. Graceful shutdowns优雅关机

Broadway boasts about having graceful shutdowns. This is a rather interesting concept to explore as it relies heavily on the concurrency system of Elixir.百老汇以优雅的关闭而自豪。这是一个相当有趣的探索概念，因为它在很大程度上依赖于 Elixir 的并发系统。

Essentially, the pipeline can only exist in two states — when all components are online and when all components are shutting down. There is no point in time where a single component will shutdown on its own without being restarted. This is because of the way that the supervisor of each component declares restart strategies ensuring that should a child process encounters any errors, it will be restarted without a hitch. This way, the only time where our components can shut down is when we shut down our main process or pipeline supervisor process. When either process is terminated, we want to properly handle all remaining events in the pipeline before shutting off every component.本质上，管道只能以两种状态存在——所有组件都在线时以及所有组件都关闭时。没有时间点可以让单个组件在不重新启动的情况下自行关闭。这是因为每个组件的主管声明重启策略的方式确保子进程遇到任何错误时，它将顺利重启。这样，我们的组件唯一可以关闭的时间是当我们关闭我们的主进程或管道主管进程时。当任一进程终止时，我们希望在关闭每个组件之前正确处理管道中所有剩余的事件。

This is achieved through a mix of concurrency features. But before we can explain how it works, a simple introduction of exit signals and process termination is due.这是通过并发特性的组合来实现的。但在我们解释它是如何工作之前，需要简单介绍一下退出信号和进程终止。

Exit signals and process termination退出信号和进程终止

Processes can be linked to one another. When either process receives an exit signal — which can occur when the process is terminated forcibly or when it receives an exit signal propagated from its parent — it will propagate the exit signal to the linked process and that process will terminate as well.流程可以相互链接。当任何一个进程接收到退出信号时——这可能发生在进程被强制终止时，或者当它接收到从其父进程传播的退出信号时——它会将退出信号传播到链接的进程，并且该进程也将终止。

However, these exit signals can be trapped instead. When this occurs, rather than terminating the process that receives the propagated exit signal, the exit signal is sent as a message, allowing the receiving process to handle the exit as though it was just another message.然而，这些退出信号可以被捕获。发生这种情况时，不是终止接收传播的退出信号的进程，而是将退出信号作为消息发送，允许接收进程处理退出，就好像它只是另一条消息一样。

When a process is terminated, an optional terminate/2 callback can be declared to perform any cleanup before the process is actually terminated. Supervisors can start a list of child processes and is responsible for managing the restart strategy of each child. The interaction between a supervisor and terminate is rather interesting. When a child is terminated, it is restarted accordingly. When a supervisor terminates, all of its children will also be terminated. If a child process traps exits, the terminate callback is called. If not, it will simply terminate immediately without calling the callback. More information about how supervisor interact with shutdowns can be found in the official documentation.主管可以启动一个子进程列表，并负责管理每个子进程的重启策略。 supervisor 和 terminate 之间的交互是相当有趣的。当一个孩子被终止时，它会相应地重新启动。当主管终止时，其所有子级也将终止。如果子进程陷入退出，则调用终止回调。如果没有，它将立即终止而不调用回调。有关主管如何与关闭交互的更多信息，请参见官方文档。

@impl true
def terminate(_, state) do
  for name <- state.first, pid = Process.whereis(name) do
    send(pid, :will_terminate)
  end

  for name <- state.producers, pid = Process.whereis(name) do
    Broadway.Topology.ProducerStage.drain(pid)
  end

  for name <- state.last, pid = Process.whereis(name) do
    ref = Process.monitor(pid)

    receive do
      {:done, ^pid} -> :ok
      {:DOWN, ^ref, _, _, _} -> :ok
    end
  end

  :ok
end

Options validation选项验证

Dashbit — the team behind Broadway — developed an options validation library called NimbleOptions that aims to be a small library for validating and documenting high-level options.Dashbit——Broadway 背后的团队——开发了一个名为 NimbleOptions 的期权验证库，旨在成为一个用于验证和记录高级期权的小型库。

A set of definitions for the available options are created first and these can be used to validate a keyword list — aka the options.首先创建一组可用选项的定义，这些定义可用于验证关键字列表——也就是选项。

If the options are invalid, an error is returned, otherwise an :ok status along with the options are returned. The returned options have default values filled in.如果选项无效，则返回错误，否则返回 :ok 状态和选项。返回的选项填充了默认值。

Default values in dictionaries字典中的默认值

Broadway has an interesting way of fanning out default values for the options keyword list. In the options keyword list, a “parent” value for :partition_by, :hibernate_after, and :spawn_opt is provided.Broadway 有一种有趣的方式来展开选项关键字列表的默认值。在选项关键字列表中，提供了 :partition_by、:hibernate_after 和 :spawn_opt 的“父”值。

The parent value will be used for producers, processors, and batchers if no explicit child value is provided. Alternatively, we might want to fan out a parent value to only two of the three unset child values while maintaining the original child value of the third.如果未提供明确的子值，则父值将用于生产者、处理器和批处理程序。或者，我们可能希望将父值扇出到三个未设置的子值中的两个，同时保持第三个的原始子值。

defp carry_over_one(opts, key, keys) do
  update_in(opts[key], fn value -> Keyword.merge(Keyword.take(opts, keys), value) end)
end

defp carry_over_many(opts, key, keys) do
  update_in(opts[key], fn list ->
    defaults = Keyword.take(opts, keys)
    for {k, v} <- list, do: {k, Keyword.merge(defaults, v)}
  end)
end
